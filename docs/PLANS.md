# Codex 执行计划（ExecPlans）

本文定义执行计划（`ExecPlan`）的要求。`ExecPlan` 是一种可执行设计文档，编码代理可以据此交付一个可工作的功能或系统变更。默认读者是这个仓库的新手：他只有当前工作树和你提供的这一份 `ExecPlan`，没有历史上下文，也不会记得之前的计划或讨论。

本仓库中的 `ExecPlan` 默认放在 `docs/exec-plans/active/`，完成或终止后移入 `docs/exec-plans/completed/`。每次完成计划并提交代码时，都必须将本次仍遗留、需要后续跟踪的技术债登记到 `docs/exec-plans/tech-debt-tracker.md`。

## 如何使用 ExecPlans 与 PLANS.md

编写可执行规格（`ExecPlan`）时，必须逐条遵循 `PLANS.md`。如果当前上下文里没有这份文件，先完整重读，再开始起草计划；起草时先从骨架模板开始，再随着研究过程补全细节。

实现可执行规格（`ExecPlan`）时，不要向用户追问“下一步做什么”，而是直接推进到下一个里程碑。在每个停顿点都要更新全部相关章节，必要时拆分或补写 `进度` 条目，明确说明已经完成了什么、剩下什么。歧义应自行消解，并在计划中留下决策记录。

讨论可执行规格（`ExecPlan`）时，所有方案调整都必须写入计划中的 `决策记录`，确保未来协作者可以明确知道“改了什么、为什么改”。`ExecPlan` 是活文档，必须做到只靠这份计划和当前工作树就能重新启动工作。

当需求难、未知项多、可行性不明确时，使用里程碑去安排原型验证、玩具实现或库级探索，以验证用户提出的方案是否可行。需要深入读依赖库源码、做局部实验、记录验证证据，再据此指导完整实现。

## 硬性要求

以下要求为硬性约束：

- 每一份 `ExecPlan` 都必须完全自包含；也就是说，以它当前的内容，就应当包含新手完成任务所需的全部知识和指令。
- 每一份 `ExecPlan` 都是活文档。随着进展推进、发现出现、设计决策落定，贡献者必须持续修订它；每次修订后仍然必须保持自包含。
- 每一份 `ExecPlan` 都必须让一个对本仓库没有先验知识的新手可以端到端完成实现。
- 每一份 `ExecPlan` 都必须导向一个可证明有效的工作结果，而不是只让代码“看起来满足定义”。
- 每一份 `ExecPlan` 都必须把术语写成普通语言；如果不能解释清楚，就不要使用该术语。

目的和意图优先。计划开头先用几句话说明这项工作为什么重要：从用户视角看，这次变更完成后能做什么、与之前相比新增了什么能力、如何亲眼验证它生效。然后再引导读者完成实现，明确写出应该改什么、跑什么命令、预期看到什么现象。

执行计划的代理可以列文件、读文件、搜索、运行项目和执行测试，但它没有你脑中的隐含上下文，也不会从前一个里程碑自动推断出你省略的意思。任何依赖的前提假设都要重复写出来。不要把关键知识外包给外部博客或文档；如果某个知识点是执行所必需的，就用你自己的话写进计划里。如果当前 `ExecPlan` 依赖另一份已入库的 `ExecPlan`，可以显式引用；如果没有入库，就必须把相关上下文完整重述在当前计划里。

## 格式要求

格式和外壳约束都很严格。每一份 `ExecPlan` 都必须是一个标记为 `md` 的单一 Markdown 代码块，以三反引号开始，也以三反引号结束。不要在这个代码块内部再嵌套三反引号代码块；如果需要展示命令、运行记录、diff 或代码片段，应当使用缩进块而不是内层代码块，以避免提前闭合外层 fence。标题使用标准 Markdown 标题语法，并在每个标题后保留两个换行。

如果把 `ExecPlan` 写入一个 `.md` 文件，并且该文件内容只包含这一份计划本身，则不需要额外包裹最外层三反引号。

正文应以 prose 为主，优先写完整句子，而不是堆砌列表。除非不用列表就会造成歧义，否则避免使用 checklist、表格或很长的枚举。只有 `进度` 章节强制使用 checklist，其余叙述性章节都应保持“正文优先”。

## 编写准则

自包含和普通语言是第一原则。如果引入一个不是日常语言的词，比如 “daemon”、“middleware”、“RPC gateway” 或 “filter graph”，必须立刻解释它的含义，并说明它在当前仓库里体现在哪里，例如对应哪个文件、哪个命令、哪个模块。不要写“见上文定义”或“参照架构文档”，需要的解释必须直接写在当前计划里，即使因此出现重复。

避免常见失败模式。不要依赖未定义术语。不要把“功能的字面要求”写得过窄，以至于最后代码虽然能编译，但没有实际意义。不要把关键决策推给读者自己做。如果存在歧义，直接在计划里解决，并解释为什么这样选。宁可多解释用户可观察到的效果，也不要把关键行为写成抽象口号。

计划必须锚定在可观察结果上。说明实现完成后用户可以做什么、需要运行什么命令、应该看到什么输出。验收标准应表述为人类可验证的行为，例如“启动服务后访问某个地址，返回 200 和某个 body”，而不是“新增了一个结构体”。如果变更本身偏内部，也要解释怎样证明它真的有效，例如某个测试在变更前失败、变更后通过，或某个端到端场景因此可用。

必须明确仓库上下文。写出准确的仓库相对路径、函数名、模块名，并说明新文件应创建在哪里。如果一次会触达多个区域，要先写一个简短的定向段，解释这些部分如何关联，帮助新手建立导航。执行命令时写清工作目录和完整命令行；如果结果依赖环境，明确说明前提和合理替代方案。

步骤必须可重试且尽量安全。写法要支持重复执行而不造成损害或漂移。如果某一步可能半途失败，说明如何重试、恢复或调整。如果确实需要迁移或破坏性操作，必须写清备份或安全回退方式。优先选择可逐步验证的增量式改动。

验证不是可选项。计划中必须写明如何运行测试、如何启动系统（若适用）、以及如何观察它做出有意义的行为。对新功能或新能力，描述足够完整的测试方式，并给出预期输出和错误信息，使新手能区分成功与失败。只编译通过不算完成；尽量通过端到端场景、CLI 调用或 HTTP 请求/响应来证明变更确实生效。

保留证据。执行步骤产生的终端输出、简短 diff 或日志，应作为缩进示例保留在单一代码块内部。证据要短、要聚焦，只保留能证明成功的部分。如果需要展示 patch，优先保留按文件分块的小段摘录，而不是贴一整坨无法消化的大补丁。

## 里程碑

里程碑是叙事工具，不是官僚负担。如果把任务拆成多个里程碑，每个里程碑都先用一小段话解释范围、结束后会新增什么、需要运行什么命令，以及预期如何验收。让整份计划读起来像一个故事：目标、工作、结果、证据。里程碑与 `进度` 是两回事：里程碑负责讲清阶段性故事，`进度` 负责跟踪细粒度状态；两者都必须存在。

每个里程碑都必须能够独立验证，并以增量方式推进整个计划的最终目标。不要为了简洁而压缩掉关键细节。

## 活文档与设计决策

- `ExecPlan` 是活文档。每当出现关键设计决策，都必须更新计划，记录决策内容和思考过程，并统一写入 `决策记录`。
- 每一份 `ExecPlan` 都必须包含并持续维护 `进度`、`意外与发现`、`决策记录` 和 `结果与复盘` 这四个章节；它们不是可选项。
- 如果在实现中发现优化器行为、性能权衡、意外 bug，或者影响方案选择的特殊语义，都要在 `意外与发现` 中记录，并附上简短证据，最好是测试输出。
- 如果中途改变方案，必须在 `决策记录` 中说明原因，并同步更新 `进度` 反映其影响。计划既是当前执行者的清单，也是下一位协作者的接手说明。
- 在重大阶段完成，或整份计划结束时，必须在 `结果与复盘` 中总结完成了什么、剩余什么以及有哪些经验教训。

## 原型里程碑与并行实现

当原型能显著降低风险时，可以而且通常应该显式加入“原型里程碑”。例如：先给依赖库补一个底层操作来验证可行性，或先探索两种组合顺序并比较优化器效果。原型应当是增量的、可测试的；要明确标注它是“原型”，说明如何运行和观察，以及什么条件下应当升格为正式实现，什么条件下应当丢弃。

优先采用“先增加、后收敛”的方式推进，并始终保持测试可运行。并行实现也可以接受，例如在迁移期间保留一个新适配层和旧路径并存，只要这样能降低风险或让测试持续通过。此时必须写清楚如何验证两条路径，以及将来如何安全地移除旧路径。如果任务涉及多个新库或多个特性区域，可以考虑先做彼此独立的 spike，分别验证每个外部库是否真的能提供所需能力。

## 优秀 ExecPlan 的骨架模板

下面是 `ExecPlan` 骨架模板；写新计划时应从这里开始，并按任务实际情况补足细节。

    # <简短、动作导向的标题>

    本 `ExecPlan` 是活文档。随着工作推进，必须持续更新 `进度`、`意外与发现`、`决策记录` 和 `结果与复盘` 这几个章节。

    如果仓库中存在 `PLANS.md`，请在这里写明该文件相对于仓库根目录的路径，并注明当前文档必须按照 `PLANS.md` 的要求持续维护。

    ## 目标与全局图景

    用几句话说明这次变更完成后，用户获得了什么能力，以及如何亲眼看到它生效。明确写出你要交付的用户可见行为。

    ## 进度

    使用带勾选框的列表记录细粒度步骤。每一个停顿点都必须在这里留下记录，即使这意味着要把一个部分完成的任务拆成两条，例如“已完成部分”和“剩余部分”。本章节必须始终反映工作的真实当前状态。

    - [x] (2025-10-01 13:00Z) 示例：已完成的步骤。
    - [ ] 示例：尚未完成的步骤。
    - [ ] 示例：部分完成的步骤（已完成：X；剩余：Y）。

    使用时间戳帮助衡量推进速度。

    ## 意外与发现

    记录实现过程中发现的异常行为、bug、优化现象或关键洞见，并附上简短证据。

    - 观察：……
      证据：……

    ## 决策记录

    按以下格式记录执行过程中做出的每一项决策：

    - 决策：……
      理由：……
      日期/作者：……

    ## 结果与复盘

    在重大里程碑完成时，或整份计划结束时，总结产出、缺口和经验教训，并将结果与最初目标对照。

    ## 上下文与定向

    假设读者一无所知，说明与当前任务相关的现状。写出关键文件和模块的完整路径，定义你将使用的非直观术语，不要引用之前的计划。

    ## 工作计划

    用正文描述修改与新增的顺序。对每一处改动，写明文件、位置（函数、模块）以及需要插入或修改的内容，要求具体且克制。

    ## 具体步骤

    写出需要执行的准确命令，以及执行目录。当命令会产生输出时，提供简短的预期记录，方便读者比对。本章节必须随着工作推进持续更新。

    ## 验证与验收

    说明如何启动或操作系统，以及应观察到什么。验收标准应写成可观察行为，明确输入与输出。如果涉及测试，可以写成：“运行 <项目测试命令>，预期 <N> 个通过；新增测试 <name> 在变更前失败，在变更后通过>”。

    ## 幂等性与恢复

    如果某些步骤可以安全重复执行，请明确写出；如果某一步有风险，提供安全的重试或回滚路径。完成后应保持环境整洁。

    ## 证据与备注

    将最重要的运行记录、diff 或片段作为缩进示例保留下来，要求简洁，并聚焦于能证明成功的内容。

    ## 接口与依赖

    这一节要有明确指令性。写明要使用哪些库、模块和服务，以及原因；说明在该里程碑结束时必须存在的类型、trait/接口和函数签名。优先使用稳定的名称和路径，例如 `crate::module::function` 或 `package.submodule.Interface`。例如：

    在 `crates/foo/planner.rs` 中定义：

        pub trait Planner {
            fn plan(&self, observed: &Observed) -> Vec<Action>;
        }

如果遵守以上要求，那么一个无状态的代理，或者一个完全陌生的人类新手，都应该能从头到尾读完 `ExecPlan` 并做出一个可工作的、可观察验证的结果。这就是标准：自包含、自足完备、面向新手、结果导向。

每次修订计划时，都必须保证变更已经同步反映到所有相关章节，包括那几个活文档章节；并且需要在计划底部写一条说明，记录这次变更是什么、为什么改。`ExecPlan` 不仅要写“做什么”，也要尽量写清“为什么这样做”。
