# BRANCH

这个文件只定义主题分支命名规则。

## 目的

- 让分支名能直接表达当前工作的范围和目标，便于 MR、review 和后续追溯。
- 保持仓库中的主题分支命名风格一致。

## 命名原则

- 一个分支名只表达一个明确目标，避免把多个无关主题揉进同一个名字。
- 命名应优先覆盖业务域、模块或文档域，再写当前主题和交付目标。
- 名称应足够具体，能让人只看分支名就知道这条分支在做什么。

## 命名格式

默认使用英文小写的 kebab-case：

```text
area-topic-goal
```

要求：

- 只使用英文小写字母、数字和 `-`。
- 不使用空格、下划线、中文或无语义缩写。

## 命名建议

- 功能或实现分支：先写受影响模块，再写功能主题和目标。
- 文档分支：优先使用 `docs-...` 开头。
- 工作流或协作规则分支：优先使用 `workflow-...` 开头。
- 修复类分支：可以直接把问题域和修复目标写进名称，不强制增加 `fix-` 前缀。

## 示例

- `mac-basic-typing-language-engine-foundation`
- `docs-mr-workflow-clarification`
- `workflow-branch-naming-rules`

## 反例

- `test`
- `update`
- `branch_for_mr`
- `修一下文档`

这些名称的问题是范围不清、不可追踪，或不符合统一格式。
