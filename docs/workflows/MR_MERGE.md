# MR_MERGE

这个文件定义 reviewer 在 MR 已完成复审后执行合并时应遵循的规则。

## 目的

- 统一 reviewer 的可合并判断口径和执行动作。
- 避免在未复审最新 head 的情况下直接合并新增提交。
- 保证进入 `main` 的版本就是已完成 review 的版本。

## 适用范围

- 默认适用于 reviewer 对已 review 的 MR 执行最终合并。
- 如果用户或团队在当前任务中明确给出不同流程，以明确要求为准。

## 可合并条件

只有在以下条件同时满足时，reviewer 才可以执行合并：

- 当前 review 未发现新的确定性 blocker。
- 已提出的 blocker 已修复，并完成对最新 MR head 的必要复审。
- 相关文档、假设记录和验证说明已经同步。
- 未解决但不阻塞的问题已经明确降级为 follow-up 或技术债，而不是继续伪装成待定状态。

## reviewer 合并动作

1. 合并前确认 MR 当前 head 就是已复审的那一版提交；如果工具支持，使用 reviewed SHA 锁定合并。
2. 确认当前 MR 的 review 结论已经明确为“未发现新的确定性 blocker”，且该结论针对的是最新 head。
3. 默认由完成最新一轮复审并确认可合并的 reviewer 执行合并动作。
4. 默认直接合入 `main`；除非团队另有明确要求，不在这里额外要求 `squash` 或 `rebase`。
5. 合并完成后默认删除远程源分支。
6. 如需继续本地开发，再把本地 `main` 更新到最新，并清理已合并的本地 / 远程跟踪分支。

## 行为边界

- 不在没有复审最新 head 的情况下直接合并。
- 不因为“差不多没问题”就跳过确定性 blocker。
- 不把尚未确认的问题包装成可合并结论。
- 不替 worker 完成本应由 worker 自己处理的实现收口、文档补齐或技术债登记。
