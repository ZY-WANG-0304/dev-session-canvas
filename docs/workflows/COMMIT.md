# COMMIT

这个文件定义提交约定。

## 目的

- 统一提交信息格式
- 保持提交说明与实际变更一致
- 避免把无关改动混入当前提交

## 与 MR 的关系

- 本文件只定义 commit 约定；MR 描述内容与格式见 `docs/workflows/MR_CREATE.md`，reviewer 的合并动作见 `docs/workflows/MR_MERGE.md`。
- 如果准备把当前提交推进成 MR，commit 主题应与 MR 的最终目标保持一致，不要在同一条分支上混入多个无关主题。

## 提交前检查

- 只暂存与当前任务直接相关的文件。
- 如果工作区存在无关改动，不要默认把它们一起提交。
- 如果用户指定了提交格式、主题或正文要求，以用户要求为准。
- 如果文档、代码和验证方式之间存在不一致，先修正文档或变更内容，再提交。

## 默认提交格式

默认使用以下格式：

```text
type(scope): 中文主题
```

要求：

- `type` 使用英文小写。
- `scope` 使用英文小写，写当前变更影响的模块、文档域或工作流范围。
- `主题` 以中文为主，直接概括本次变更。

## 类型

默认从以下类型中选择：

- `feat`：新增功能（feature）
- `fix`：修复 bug（bug fix）
- `docs`：文档更新（documentation）
- `style`：代码格式调整，不影响运行，例如空格、缩进等
- `refactor`：代码重构，不属于功能新增或 bug 修复
- `test`：测试相关变动，包括新增或修改测试
- `chore`：构建工具或辅助流程变动，例如依赖更新
- `other`：其他增强性改动，不属于功能新增或问题修复
- `revert`：回滚提交

## 提交正文

如果需要正文，默认使用中文编号列表说明主要变更：

```text
1. 说明第一项关键变更。
2. 说明第二项关键变更。
3. 说明第三项关键变更。
```

要求：

- 只写当前提交实际包含的内容。
- 以变更结果为主，不写空泛描述。
- 如果本次提交只包含单一且清晰的变更，可以不写正文。
- 多行正文必须使用真实换行，不要把 `\n` 作为字面量写入 commit message。

## 范围选择

- 文档体系调整优先使用 `docs(...)`
- 某个产品或模块变更优先使用对应模块名
- 纯工作流或协作约定调整，可使用 `workflow`

## 用户优先

- 用户明确给出提交模板、示例或格式要求时，覆盖本文件默认规则。
- 用户要求中文为主时，主题和正文都应以中文为主。
- 用户要求只提交部分文件时，不要擅自扩大提交范围。
