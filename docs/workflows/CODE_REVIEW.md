# CODE_REVIEW

本文件定义本仓库进行远端 MR / 本地变更 Code Review 时的核心规则。目标是保护主路径质量，同时避免把推测性问题误报成阻塞项。

## 阻塞项标准

只有以下问题应作为“必须修复后再合并”的 review finding：

- 已能稳定复现，或可从当前代码路径直接证明的功能 bug / regression。
- 会影响当前主路径可用性、数据正确性、安全性或正式验收口径的重大风险。
- 当前 MR 引入且已经确认的文档漂移：正式产品、设计、架构或 `ExecPlan` 文档与实现或仓库事实矛盾，并足以误导后续实现或验收。

不要把 blocker 建立在“可能”“也许”“未来可能会出问题”上。若结论依赖未验证的系统行为、框架内部调用路径、线上环境、偶发现象或不稳定复现，默认不作为 blocker。

## 非阻塞项与技术债

以下问题默认不阻塞当前 MR：

- 测试偶现、环境抖动或暂时无法稳定复现的问题。
- 需要额外系统级验证才能确认的风险点。
- 当前原型边界内已知但尚未证明会稳定影响主路径的行为。
- 风格偏好、命名建议、结构整理或“顺手重构”类意见。

这类问题应进入 `docs/exec-plans/tech-debt-tracker.md`，而不是继续阻塞当前 MR。登记时至少写清：

- 背景与触发条件
- 影响范围
- 当前临时处理或边界
- 建议修复时机
- 关联文档或代码路径

## 评论写法

每条 blocker 评论都应尽量包含以下信息：

1. 严重级别
2. 文件与位置
3. 问题本身，以及为什么它是确定性的
4. 复现步骤、观察证据，或可以直接成立的代码路径分析
5. 期望的修复方向

Review 总结顺序固定为：

1. 先列 findings，并按严重度排序
2. 如果没有 blocker，明确写“未发现新的确定性 blocker”
3. 如有残余风险，单列为 tech debt / follow-up，不与 blocker 混写

## Reviewer 行为边界

- 不把未确认内容写成已确认事实。
- 不把个人偏好或更“优雅”的方案写成必须修复的问题。
- 不要求 worker 顺手解决与当前 MR 无直接关系的历史问题。
- reviewer 默认通过 comment 提出问题，由 worker 负责修复、登记技术债并更新 MR。
- reviewer 必须基于最新的 MR head 复审，不能跳过 worker 的新增提交直接给出可合并结论。
- 在仓库默认工作流里，当 reviewer 确认最新 head 未发现新的确定性 blocker 后，由 reviewer 执行合并；但 reviewer 不直接代 worker 补技术债文档或替 worker 完成实现收口。

## 与仓库文档的关系

- Review 以当前正式文档为准，尤其是 `AGENTS.md`、`ARCHITECTURE.md`、`docs/PLANS.md`、`docs/product-specs/` 和 `docs/design-docs/`。
- 若实现与正式文档冲突，需要明确指出究竟是代码错误、文档错误，还是二者都需要一起收口。
- 对复杂功能或重构，计划完成后仍遗留的问题必须按 `docs/PLANS.md` 要求登记到 `docs/exec-plans/tech-debt-tracker.md`。
