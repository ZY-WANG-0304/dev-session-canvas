# CODE_REVIEW

本文件定义本仓库进行远端 MR 或本地变更 Code Review 时的核心规则。目标是保护主路径质量，同时避免把推测性问题误报成阻塞项。

## 阻塞项标准

只有以下问题应作为“必须修复后再合并”的 finding：

- 已能稳定复现，或可从当前代码路径直接证明的功能 bug / regression
- 会影响当前主路径可用性、数据正确性、安全性或正式验收口径的重大风险
- 当前 MR 引入且已确认的文档漂移，足以误导后续实现或验收

不要把 blocker 建立在“可能”“也许”或“未来可能出问题”上。

## 非阻塞项与技术债

以下问题默认不阻塞当前 MR：

- 测试偶现、环境抖动或暂时无法稳定复现的问题
- 需要额外系统级验证才能确认的风险点
- 当前范围外的结构整理或顺手重构建议
- 风格偏好或命名偏好

这类问题应登记到 `docs/exec-plans/tech-debt-tracker.md`，而不是继续阻塞当前 MR。

## 评论写法

每条 blocker 评论都应尽量包含以下信息：

1. 严重级别
2. 文件与位置
3. 问题本身，以及为什么它是确定性的
4. 复现步骤、观察证据，或可直接成立的代码路径分析
5. 期望的修复方向

Review 总结顺序固定为：

1. 先列 findings，并按严重度排序
2. 如果没有 blocker，明确写“未发现新的确定性 blocker”
3. 如有残余风险，单列为 tech debt / follow-up

## Reviewer 行为边界

- 不把未确认内容写成已确认事实
- 不把个人偏好写成必须修复的问题
- 不要求 worker 顺手解决与当前 MR 无直接关系的历史问题
- 必须基于最新 MR head 复审，不能跳过新增提交直接给出可合并结论

## 与仓库文档的关系

- Review 以当前正式文档为准，尤其是 `AGENTS.md`、`ARCHITECTURE.md`、`docs/PLANS.md`、`docs/product-specs/` 和 `docs/design-docs/`
- 若实现与正式文档冲突，需要明确指出究竟是代码错误、文档错误，还是二者都需要一起收口
