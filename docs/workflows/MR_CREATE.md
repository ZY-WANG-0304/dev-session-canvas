# MR_CREATE

这个文件只定义 MR 描述应包含的信息和默认格式。

## 目的

- 让 reviewer 和后来者能从 MR 页面快速理解这次变更要解决什么问题。
- 保证 MR 描述与实际代码、文档和验证证据一致。

## 适用范围

- 默认适用于创建新 MR 时编写描述，以及后续更新 MR 时同步描述。
- 如果用户或团队在当前任务中明确给出不同模板，以明确要求为准。

## 描述内容

MR 描述默认至少包含以下内容：

- 背景 / 目标：这次变更要解决什么问题，为什么值得 review。
- 主要改动：本 MR 实际改了哪些模块、文档或脚本。
- 验证方式：跑了哪些测试、脚本或手动验证，结果是什么；如果本轮无法验证，要直接写明原因。
- 残余风险：还有哪些已知但不阻塞当前合并的问题。

## 默认格式

默认建议使用以下结构：

```md
## 背景 / 目标

## 主要改动

## 验证方式

## 残余风险
```

如果当前 MR 很小，可以写得更短；但最终仍应让后来者能从 MR 页面追溯清楚“改了什么、怎么验证、还剩什么风险”。

## 写法要求

- 只写当前 MR 实际包含的范围，不把无关历史问题堆进描述里。
- 未确认内容不要写成已确认结论；如果仍有边界或风险，明确写出当前状态。
- 验证方式只写实际执行过的内容，不要把计划中的验证写成已经完成。
- 残余风险只写已知且当前不阻塞合并的问题；如果已有对应技术债或正式文档，可直接写明位置。
