# PRODUCT_SENSE

本文档用于统一当前项目在产品方向、优先级和体验取舍上的判断。当具体需求文档没有覆盖决策细节时，应优先参考本文件。

## 产品主张

本项目是一款 VSCode 插件。
它的定位是面向 VSCode 的多 Agent 协作画布。核心目标不是把独立 app 的全部能力搬进编辑器，而是为 `Agent` 与 `Terminal` 提供一个共享的全局视角，并与 VSCode 现有插件生态协同，提升 AI 开发时代的开发体验。

## 目标用户与非目标用户

当前阶段优先服务已经在 VSCode 中进行 AI 协作的开发者。这类用户更在意：

- 多个 `Agent` 与终端会话能否放在同一视图中理解
- 当前 workspace 上下文能否自然融入协作流程
- 现有编辑器、终端和插件能力能否继续直接使用

当前阶段不优先服务以下诉求：

- 需要完整独立桌面应用体验的用户
- 主要关心工作区管理、启动器或多 workspace 管理的用户
- 只需要单轮聊天面板、不需要空间化协作视图的用户

## 核心问题

当开发者在一个工程里并行使用多个 Agent 与终端会话时，上下文往往被分散在多个面板、标签页和工具窗口中，导致用户难以持续看清：

- 现在有哪些执行单元正在运行
- 它们分别在做什么
- 它们与当前代码、工具和插件上下文如何关联
- 当前整体任务推进到了哪里

## 核心场景

产品应优先服务以下主路径：

- 用户在一个 VSCode workspace 中打开画布。
- 用户在同一画布上放置并观察多个 `Agent` 与 `Terminal`。
- 用户在多对象并行活动时仍能维持全局视角，并快速回到某个局部上下文继续工作。
- 用户继续配合 VSCode 现有编辑器、终端和插件生态完成开发，而不是被迫迁移到一套封闭工作台。

## 价值主张与差异化

当前产品的核心价值不在于对象种类越多越好，而在于把原本分散的 `Agent` 与终端会话重新放回同一个空间化视图中，并与 VSCode 现有插件生态配合，让用户获得：

- 对 `Agent` / `Terminal` 更强的全局可见性
- 更低的上下文切换成本
- 更自然的多 Agent 协作组织方式
- 不脱离现有 VSCode 开发习惯的 AI 开发增强体验

## 当前阶段判断

- 自 2026-04-05 起，项目已完成 MVP 验证；后续工作默认在已验证主路径上做正式开发、质量收口和持续迭代。
- 文档中的 `MVP` 主要表示当前范围优先级与验收基线，不再表示“只做原型、不要求长期维护”。
- 复刻优先级应放在核心产品体验，而不是独立 app 的外围能力。
- workspace 管理不属于当前阶段的高优先级，因为 VSCode 已经提供基础 workspace 语义。
- 当前更重要的是先闭合“同一张画布上的 `Agent` / `Terminal` 全局视角主路径”，再扩展其它能力。
- 当前阶段允许借助 VSCode 原生能力闭合主路径，而不是要求所有体验从第一天起都完全嵌入同一张画布。
- `Note` 可以作为轻量辅助协作对象存在，但不是产品成立的前提。
- 当前 UI 风格应定义为“跟随 VSCode 主题的极简协作画布”，而不是固定深色画布。

## 视觉语言规则（Style Brief）

当前产品的视觉语言应被定义为：`VSCode 原生语境下、主题跟随的极简工具型协作画布`。

这句话的含义不是“界面尽量素”，而是要求所有可见层设计都优先服务以下感受：

- 它首先像 VSCode 中一个可信、克制、可长期使用的原生工作面，而不是额外叠加在编辑器上的营销式 dashboard。
- 它首先帮助用户看清当前有哪些对象、哪些对象正在运行、哪里可以继续工作，而不是先追求品牌化氛围或强风格装饰。
- 它首先是一张承载工作上下文的工具型画布，而不是一张独立 app 风格的沉浸式深色白板。

当前阶段的具体视觉规则如下：

### 主题规则

- 界面必须跟随 VSCode 当前主题工作，浅色主题和深色主题都应成立。
- 颜色优先来自 `--vscode-*` token，而不是写死一套独立的产品底色。
- 不允许把“深色画布”当成默认产品身份；深色只应来自用户当前主题，而不是组件自身的固定偏置。

### 画布基调

- 画布应表现为低噪音、可长期停留的工作底面，视觉重心应让位于节点本体。
- 背景可以有网格或轻微层次，但只能帮助定位和导航，不能抢走对象的可读性。
- 固定角落控件应克制，避免把画布重新做成带大块 chrome 的面板页。

### 节点表面语言

- `Agent` 和 `Terminal` 应明确表现为运行中的工作窗口，而不是抽象卡片。
- `Note` 应表现为轻量工作文档或上下文纸面，而不是半套任务系统或配置面板。
- 不同对象可以有类型差异，但必须共享同一套总体语言：窗口化、工具化、低噪音。
- 类型差异优先通过标题、副标题、边框、状态胶囊和少量强调色表达，而不是依赖大面积内部染色。

### 色彩与层级规则

- 颜色的主要职责是表达状态、优先级和可操作性，不是制造风格存在感。
- 避免在节点内部堆叠大面积语义色块、渐变块或“卡片套卡片”式层级。
- 当颜色语言和信息可读性冲突时，优先可读性；当类型区分和整体一致性冲突时，优先整体一致性。

### 表单感规则

- 只有当对象本质上真的是“填写结构化字段”时，才允许出现明显表单感。
- 当前主路径中的节点不应默认长得像 inspector 或配置面板。
- 对于 `Note` 这类辅助对象，应优先呈现为内容面，而不是字段集合。

### 外层控件与宿主关系

- 能放回 VSCode 宿主层的全局动作，应优先放回宿主层，而不是继续压在画布上。
- 画布内只保留和导航、选择、对象操作直接相关的最小必要 UI。
- 整体观感应更接近“VSCode 里的一个稳定工作面”，而不是“VSCode 里嵌了另一个完整产品前台”。

### 排除项

- 不追求独立 app 式品牌氛围、强装饰感 hero、固定深色渐变背景或高饱和类型染色。
- 不追求看起来功能很多的 dashboard 感。
- 不为了显得“完整”而把大量说明文字、统计卡片或辅助控制常驻在画布上。

### 取舍原则

- 当“风格表达”和“VSCode 原生适配”冲突时，优先 VSCode 原生适配。
- 当“视觉个性”和“全局可读性”冲突时，优先全局可读性。
- 当“节点类型差异”与“整张画布的一致工作语境”冲突时，优先整张画布的一致语境。
- 当“强语义色彩”与“浅色主题下的自然成立”冲突时，优先浅色主题下也自然成立。

## 产品原则

- 当功能数量和全局可读性冲突时，优先全局可读性。
- 当完美复刻和 VSCode 语境适配冲突时，优先适配 VSCode。
- 先让 `Agent` 与 `Terminal` 的核心协作闭环成立，再决定哪些辅助对象值得保留或增强。
- 优先与 VSCode 已有编辑器、终端和插件生态协同，而不是重复造一套封闭工具链。
- 不为了看起来完整而过早引入复杂管理系统。
- 当“画布内沉浸感”和“实现风险/恢复能力”冲突时，优先保证主路径成立与状态可恢复。

## 非目标

- 当前阶段不做独立 app 的 workspace 管理复刻。
- 当前阶段不追求对参考产品的像素级或所有功能级复制。
- 当前阶段不把所有可能的协作对象都纳入第一版主路径，也不要求 `Note` 成为产品成立前提。
- 当前阶段不以纯浏览器形态或宿主无差别兼容为前提成立全部核心体验。
- 当前阶段不试图替代 VSCode 丰富的插件生态。

## 成功信号

当前阶段的核心成功信号是：用户能在 VSCode 内通过这张画布管理多 Agent 协作，并在多数时间里保持对 `Agent` / `Terminal` 整体状态的可见性。

这一定义还隐含两条体验要求：

- 用户可以在画布与 VSCode 原生执行能力之间顺畅往返，而不会失去上下文。
- 在常见开发场景下，包括 reload、重开窗口或远程工作区，关键上下文不会轻易丢失。

## 假设与风险

- 当前核心假设是：空间化画布确实能显著降低多 Agent 协作时 `Agent` / `Terminal` 上下文的丢失。
- 当前主要风险是：如果画布只增加视觉复杂度，却没有显著提升全局可见性，产品体验就不能成立。
- 另一个关键风险是：如果产品与 VSCode 现有编辑器和插件生态割裂，用户不会愿意把它作为日常开发主路径的一部分。
