# PRODUCT_SENSE

本文档用于统一当前项目在产品方向、优先级和体验取舍上的判断。当具体需求文档没有覆盖决策细节时，应优先参考本文件。

## 产品主张

本项目是一款 VSCode 插件。
它的核心目标不是把独立 app 的全部能力搬进编辑器，而是在 VSCode workspace 内复刻 OpenCove 的核心协作体验：把 AI Agents、终端、任务和笔记放到同一张无限 2D 画布上。

## 目标用户与非目标用户

当前阶段优先服务已经在 VSCode 中进行 AI 协作或 Harness Engineering 的开发者。这类用户更在意：

- 多个执行单元能否放在同一视图中理解
- 当前 workspace 上下文能否自然融入协作流程
- 任务、终端、笔记和 Agent 之间的关系是否清楚

当前阶段不优先服务以下诉求：

- 需要完整独立桌面应用体验的用户
- 主要关心工作区管理、启动器或多 workspace 管理的用户
- 只需要单轮聊天面板、不需要空间化协作视图的用户

## 核心问题

当开发者在一个工程里并行使用多个 Agent、终端、任务和笔记时，上下文往往被分散在多个面板、标签页和工具窗口中，导致用户难以持续看清：

- 现在有哪些执行单元正在运行
- 它们分别在做什么
- 这些对象之间是什么关系
- 当前整体任务推进到了哪里

## 核心场景

产品应优先服务以下主路径：

- 用户在一个 VSCode workspace 中打开画布。
- 用户在同一画布上放置并观察 Agent、终端、任务和笔记。
- 用户在多对象并行活动时仍能维持全局视角，并快速回到某个局部上下文继续工作。

## 价值主张与差异化

当前产品的核心价值不在于对象种类越多越好，而在于把原本分散的协作单元放回同一个空间化视图中，让用户获得：

- 更强的全局可见性
- 更低的上下文切换成本
- 更自然的多 Agent 协作组织方式

## 当前阶段判断

- 复刻优先级应放在核心产品体验，而不是独立 app 的外围能力。
- workspace 管理不属于当前阶段的高优先级，因为 VSCode 已经提供基础 workspace 语义。
- 当前更重要的是先闭合“同一张画布上的多对象协作主路径”，再扩展其它能力。
- 当前阶段允许借助 VSCode 原生能力闭合主路径，而不是要求所有体验从第一天起都完全嵌入同一张画布。

## 产品原则

- 当功能数量和全局可读性冲突时，优先全局可读性。
- 当完美复刻和 VSCode 语境适配冲突时，优先适配 VSCode。
- 先让 Agent、终端、任务、笔记四类核心对象形成闭环，再扩展外围对象或管理能力。
- 不为了看起来完整而过早引入复杂管理系统。
- 当“画布内沉浸感”和“实现风险/恢复能力”冲突时，优先保证主路径成立与状态可恢复。

## 非目标

- 当前阶段不做独立 app 的 workspace 管理复刻。
- 当前阶段不追求对参考产品的像素级或所有功能级复制。
- 当前阶段不把所有可能的协作对象都纳入第一版主路径。
- 当前阶段不以纯浏览器形态或宿主无差别兼容为前提成立全部核心体验。

## 成功信号

当前阶段的核心成功信号是：用户能在 VSCode 内通过这张画布完成多 Agent 协作，并在多数时间里保持对整体工作的可见性。

这一定义还隐含两条体验要求：

- 用户可以在画布与 VSCode 原生执行能力之间顺畅往返，而不会失去上下文。
- 在常见开发场景下，包括 reload、重开窗口或远程工作区，关键上下文不会轻易丢失。

## 假设与风险

- 当前核心假设是：空间化画布确实能显著降低多 Agent 协作时的上下文丢失。
- 当前主要风险是：如果画布只增加视觉复杂度，却没有显著提升全局可见性，产品体验就不能成立。
- 另一个关键风险是：VSCode 的宿主边界可能限制某些独立 app 中自然成立的体验，需要重新设计而不是照搬。
