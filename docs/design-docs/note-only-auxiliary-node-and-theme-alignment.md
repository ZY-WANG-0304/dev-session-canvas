# 辅助节点收口为 Note 与主题跟随表面设计

## 1. 背景

当前画布已经把执行型对象收口为 `Agent` 与 `Terminal` 的 runtime window，并且最近一轮把 `Task` / `Note` 改成了节点内可编辑的窗口化表面。

但基于最新体验判断，当前实现还存在几条新的收口需求：

- 当前产品的 UI 风格应定义为“跟随 VSCode 主题的极简协作画布”，而不是把深色渐变与固定暗色混色当成产品风格本体。
- `Task` 与 `Note` 当前并没有形成清晰分工；继续同时保留两个辅助对象，只会让对象模型、创建入口和视觉语言变得更重。
- `Note` 虽然已经收口为单一辅助对象，但当前头部与正文仍有多余层级，例如 `Note` 类型标签、状态胶囊、kicker 和底部 hint。
- `Agent`、`Terminal`、`Note` 的标题编辑位置仍不一致，且左下角导航控件还没有完全回到 VSCode 原生工具条语言。

因此，本轮不再继续给 `Task` 增加能力，而是先把当前辅助对象收口为单一 `Note`，并顺手把节点表面与画布底色继续拉回“主题跟随、弱色块、低层级噪音”的方向。

## 2. 问题定义

本轮要回答六个问题：

1. 当前主画布中，哪些节点类型仍然属于正式支持范围。
2. `Task` 与 `Note` 没有稳定语义区分时，是否还应继续同时保留。
3. 当前节点内部的橙/紫大色块、卡片套卡片层级，是否破坏了与执行型窗口的一致性。
4. 浅色 VSCode 主题下，当前深色偏置是否还能被视为“原生适配”。
5. `Note` 的标题和正文结构还能否继续收口，避免保留已经没有必要的二级层级。
6. 标题编辑与左下角 controls 是否也应进一步对齐 VSCode 原生工作面语言。

## 3. 目标

- 把当前正式支持的节点集合收口为 `Agent`、`Terminal`、`Note`。
- 移除 `Task` 的创建、渲染、持久化和测试覆盖，不再把它描述为当前产品对象。
- 让 `Note` 成为单一轻量辅助对象，保持节点内编辑，但不再承担“与 Task 做功能分工”的额外 UI 负担。
- 把画布与节点表面继续收口到主题跟随的 VSCode 原生语境，降低固定深色混色与内部大面积语义染色。
- 让三类节点都能直接在标题栏编辑标题，形成一致的对象命名主路径。

## 4. 非目标

- 不在本轮重新引入新的辅助对象来替代 `Task`。
- 不在本轮为 `Note` 增加复杂富文本、块编辑或任务系统能力。
- 不在本轮重做执行型节点的运行模型。
- 不在本轮承诺所有历史持久化状态都要自动迁移回一个等价的 `Note`。

## 5. 候选方案

### 5.1 保留 Task 与 Note，只做轻量视觉收口

优点：

- 代码改动较小。
- 不需要修改对象模型与测试矩阵。

不选原因：

- 当前用户已经明确判断 `Task` 与 `Note` 没有形成实际功能分工。
- 即便只做视觉收口，创建入口、协议和文档仍然要继续解释两个辅助对象的差异，噪音没有实质降低。

### 5.2 保留 Task，但把 Note 删除

优点：

- `Task` 自带状态字段，看起来更像“结构化协作对象”。

不选原因：

- 当前用户反馈的真实使用路径更接近“记录工作上下文”，而不是在画布里维护轻量任务系统。
- `Task` 的状态、负责人和描述字段会自然引入更强表单感，与“轻量辅助对象”目标相冲突。

### 5.3 删除 Task，只保留 Note 作为当前辅助对象，并把视觉语言继续收口到主题跟随窗口表面

优点：

- 直接减少对象模型复杂度。
- 让辅助对象语义回到“记录上下文”，不再在画布里引入半套任务面板。
- 可以顺势删除最强的内部语义色块与分层噪音，让辅助对象更像与执行型节点同体系的轻量工作窗口。

风险：

- 需要同步修改协议、宿主状态、侧栏文案、Webview 和自动化测试。
- 需要明确旧 `task` 持久化状态的处理边界。

## 6. 风险与取舍

- 取舍：当前直接删除 `Task`，而不是先把它隐藏为实验能力。
  原因：如果对象模型已确认要收口，继续保留隐藏开关只会让实现、测试和文档长期分叉。

- 风险：已有 workspace 状态里可能存有 `task` 节点。
  当前缓解：宿主在读取旧状态时直接过滤掉 `task` 节点，并把这条边界明确写入文档；当前不承诺自动迁移为 `Note`。

- 风险：删除 `Task` 后，`Note` 会不会再次变成“什么都能装”的杂项对象。
  当前缓解：本轮只把 `Note` 定义为轻量工作笔记，不新增状态、负责人等结构化字段。

- 风险：如果只删 `Task`，但继续保留深色偏置和大面积内部染色，用户仍会感到表面语言不统一。
  当前缓解：本轮同时收口主题跟随和弱色块策略，把辅助节点内部视觉重心降到内容面本身。

## 7. 当前结论

当前已选定以下路线：

### 7.1 当前正式支持的节点只保留三类

- 执行型对象：`Agent`、`Terminal`
- 轻量辅助对象：`Note`

`Task` 不再属于当前正式支持范围；创建入口、文档、协议、测试与持久化读取都应按此收口。

### 7.2 旧 Task 状态不做等价迁移

- 宿主读取旧持久化状态时，如果遇到 `task` 节点，直接过滤掉，不再回到当前对象图。
- 当前不承诺把旧 `task` 自动转换为 `note`，避免误造伪等价语义。

### 7.3 Note 继续保留节点内编辑，但收口为更轻的工作窗口

- `Note` 保留标题和正文两项核心编辑能力。
- `Note` 的节点头部继续复用 `window-chrome`，但标题区域直接显示并编辑 `Note N` 这类标题，不再保留单独的 `Note` 类型标签行。
- `Note` 标题栏右侧不再显示状态胶囊，只保留删除动作。
- 正文区应表现为单一主内容面，只保留一个负责写入和显示正文的主要文本区域，而不是“上下两块强染色卡片”或额外的二级说明块。
- 不再为了和 `Task` 区分而保留额外语义色块、伪任务信息或多层嵌套块。

### 7.4 主题跟随高于深色偏置

- 产品风格定义为“跟随 VSCode 主题的极简协作画布”。
- 固定深色混色只能作为有限装饰，不能让浅色主题仍明显呈现深色底。
- 节点和画布优先使用 VSCode 主题 token；必要的类型区分优先放在边框、状态胶囊和轻量强调上，而不是内部大面积色块。

### 7.5 标题编辑与导航控件也应收口到原生工作面语言

- `Agent`、`Terminal`、`Note` 三类节点都允许在标题栏直接改名；标题属于对象身份的一部分，应留在对象头部，而不是分散在正文区或只对单一节点类型开放。
- `Agent` 与 `Terminal` 保留当前副标题、provider 与运行控制，但标题本身不再是只读文本。
- 左下角 controls 应表现为接近 VSCode 原生工具控件的低强调按钮组，而不是高圆角、玻璃感较强的悬浮卡片。

## 8. 验证方法

至少需要完成以下验证：

1. 侧栏与创建入口中不再出现 `Task`。
2. 新建对象时只剩 `Agent`、`Terminal`、`Note` 三类。
3. 旧状态中含 `task` 节点时，当前宿主不会因无法识别而报错，且 `task` 不会重新出现在画布里。
4. `Note` 在视觉上不再呈现明显的“双层染色卡片”结构，而更接近执行型节点同体系的轻量窗口。
5. 在浅色与深色 VSCode 主题下，画布和 `Note` 节点都不应强行保留明显深色底色偏置。
6. `Agent`、`Terminal`、`Note` 的标题栏都可以直接改名，并在 reload 后恢复。
7. 左下角 controls 在视觉上不再像自定义玻璃浮层，而更接近 VSCode 原生工具条。

## 9. 已完成实现与验证

当前实现已经按上述结论落地：

- `Task` 已从当前对象模型、创建入口、持久化恢复与自动化测试矩阵中移除。
- `Note` 标题栏现在直接显示并编辑节点标题，右侧只保留删除动作；正文区只保留一个写入和显示正文的文本区域。
- `Agent`、`Terminal`、`Note` 三类节点都通过统一的 `webview/updateNodeTitle` 链路在标题栏直接改名。
- 左下角 controls 已改为更平直、低强调的按钮组视觉，优先复用 VSCode 主题 token，而不是维持玻璃感较强的悬浮卡片。

本轮验证结果：

1. `npm run typecheck` 通过。
2. `npm run build` 通过。
3. `npm run test:webview` 通过；期间同步更新了新的画布截图基线。
4. `npm run test:smoke` 通过，含 trusted / restricted workspace 两条真实 VS Code 宿主路径。
