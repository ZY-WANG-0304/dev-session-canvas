# Task 与 Note 可编辑节点设计

## 1. 背景

当前原型里，`Task` 和 `Note` 仍然只是用于占位的静态卡片。它们能验证节点布局，却无法验证“画布中的非执行型对象”是否真的能承载可编辑协作上下文。

这会导致当前原型失去两个关键验证：

- 任务和笔记的真实内容能否进入宿主权威状态并在 reload 后恢复
- 执行型对象之外的协作对象，是否也能在画布中直接完成最小编辑闭环

## 2. 问题定义

本轮需要回答的问题是：

1. `Task` 和 `Note` 第一版应至少支持哪些字段，才能不再只是占位卡片。
2. 编辑中的局部输入状态与宿主已提交状态，应如何分层。
3. 第一版是继续走 inspector 驱动，还是在节点内部直接编辑。

## 3. 目标

- 让 `Task` 节点至少支持标题、状态、任务描述和负责人编辑。
- 让 `Note` 节点至少支持标题和正文编辑。
- 保持“宿主权威状态 + 节点内本地编辑态”的当前运行时主线。

## 4. 非目标

- 不在本轮实现富文本、Markdown 预览或块编辑器。
- 不在本轮实现任务与 Agent 的自动引用联动。
- 不在本轮引入多人协作、冲突合并或版本历史。

## 5. 候选方案

### 5.1 继续使用占位卡片

不选原因：

- 无法验证真实内容编辑、持久化和恢复链路。
- 会让画布中的非执行型对象长期停留在“看起来存在，但实际上不能用”的阶段。

### 5.2 使用右侧 inspector 编辑 Task/Note

优点：

- 节点本体更轻，编辑表单可以更完整。

不选原因：

- 会把本应在画布节点内完成的最小编辑动作再次抽离出去。
- 已有 Agent 路线已经证明，过度依赖 inspector 很容易把产品语义带偏。

### 5.3 在节点内部直接编辑

特点：

- 节点本体就是最小编辑面。
- 用户可在画布上直接修改标题、正文或状态。

优点：

- 与“把所有协作对象放到同一张画布上直接理解和操作”的目标一致。
- 能直接验证输入、提交、恢复和状态投影的真实链路。

风险：

- 输入控件会带来焦点、组合输入法和 React Flow 重渲染问题。
- Task/Note 节点尺寸会比占位卡片更大。

## 6. 当前结论

当前收敛结论如下：

- `Task` 和 `Note` 第一版都改为节点内直接编辑。
- 编辑态采用“节点本地输入状态 + 宿主已提交状态”的双层模型：
  - 输入过程由节点本地状态负责，避免被父级重渲染打断
  - 已提交内容由宿主权威状态持有并持久化
- 最小字段定义：
  - `Task`：标题、状态、任务描述、负责人
  - `Note`：标题、正文

## 7. 风险与取舍

- 取舍：第一版只做纯文本和简单状态，不做富文本。
  原因：当前优先验证对象可编辑和可恢复，不是先堆复杂编辑器能力。

- 风险：输入控件可能再次触发焦点丢失或中文输入法问题。
  当前缓解：明确不让宿主状态参与实时输入控制，实时输入只留在节点本地状态。

- 风险：如果 Task/Note 节点都变大，画布密度会下降。
  当前缓解：保持字段最小化，只提供验证闭环所需的编辑面。

## 8. 验证方法

至少需要完成以下验证：

1. 用户可以在 Task 节点内直接编辑标题、状态、任务描述和负责人。
2. 用户可以在 Note 节点内直接编辑标题和正文。
3. 已提交内容在 Webview reload 后仍可恢复。
4. 输入过程中不会因为节点重渲染而立刻丢失焦点或清空内容。

## 9. 当前验证状态

- 设计结论已完成收口。
- `npm run build` 已通过。
- `npm run typecheck` 已通过。
- 用户已完成手动验证，确认 Task 标题、状态、任务描述、负责人，以及 Note 标题、正文都可在节点内正常编辑并提交。
- 用户已完成失焦场景验证，确认点击画布其他区域后，已输入内容不会回退到旧值。
- 用户已完成 reload 场景验证，确认已提交内容可以恢复。
