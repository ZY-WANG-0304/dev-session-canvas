# 画布节点删除设计

## 1. 背景

当前原型已经支持在画布中创建四类核心节点，也已经把 `Agent` 和 `Terminal` 收敛到节点内运行的会话窗口，把 `Task` 和 `Note` 收敛到节点内编辑。

但对象生命周期还停在“只能新增，不能移除”。一旦用户误建节点、结束某项任务，或者需要清理画布，只能依赖“重置整个宿主状态”。这既不符合真实协作画布的基本预期，也阻断了我们对“对象是否能被安全销毁、执行会话是否会被同步清理、删除后恢复边界是否正确”的验证。

## 2. 问题定义

本轮需要回答的问题是：

1. 删除入口应该放在哪里，才能既显式又不打断画布上的工作流。
2. 删除执行型节点时，宿主应该如何处理活跃会话，才能避免向已删除节点继续回写状态。
3. 删除是否应当允许画布变成空白，以及 reload 后恢复边界该如何定义。
4. 当前基于数组长度生成节点 ID 的方式，在允许删除后如何保持唯一性。

## 3. 目标

- 让 `Agent`、`Terminal`、`Task` 和 `Note` 四类节点都具备最小可用的删除路径。
- 保证删除执行型节点时，会话、监听器、定时同步与宿主状态一起被清理。
- 让删除结果在 reload 或重新打开画布后仍然成立，不被默认节点回填覆盖。
- 保持节点 ID 唯一，不因删除中间节点而复用仍然存活的旧 ID。

## 4. 非目标

- 不在本轮实现删除确认弹窗、撤销/重做或历史回放。
- 不在本轮实现多选批量删除。
- 不在本轮引入“归档”“软删除”或回收站。

## 5. 候选方案

### 5.1 只在右上角控制面板提供删除当前选中节点

优点：

- UI 集中，节点头部不再增加操作控件。

不选原因：

- 删除动作离节点本体过远，用户需要先选中再移动视线到面板，效率过低。
- 对执行型节点来说，用户最自然的心智模型是“在窗口本体上关闭这个对象”，而不是去外部面板找入口。

### 5.2 只提供节点头部按钮

优点：

- 删除入口最显式，和节点本体强绑定。

不选原因：

- 只提供鼠标路径会让键盘流断裂。
- 当用户已经选中节点但鼠标停在别处时，仍然缺少快速删除方式。

### 5.3 节点头部按钮 + 选中节点键盘删除

特点：

- 节点头部始终提供显式删除按钮。
- 当前有选中节点时，允许使用 `Delete` / `Backspace` 删除。
- 当焦点位于输入框、下拉框、按钮等交互控件上时，不触发画布级删除快捷键。

当前结论：采用此方案。

## 6. 当前结论

当前收敛结论如下：

- 删除能力同时提供两条路径：
  - 每个节点头部提供统一删除按钮；
  - 当前有选中节点且焦点不在交互控件内时，允许用键盘删除。
- 删除动作直接作用于“对象生命周期”：
  - `Task` / `Note` 删除时，宿主直接移除节点并落盘；
  - `Agent` / `Terminal` 删除时，宿主先销毁活跃会话，再移除节点并落盘。
- 删除后允许画布为空。初次打开且没有存储时仍生成默认示例节点；但如果用户已经显式删除全部节点并持久化为空数组，reload 后必须保持空画布。
- 节点创建序号不再依赖数组长度，而改为根据当前节点集合计算下一个未使用的序号，避免删除中间节点后复用已有 ID。

## 7. 风险与取舍

- 取舍：第一版不做删除确认。
  原因：当前更重要的是先闭合对象生命周期与宿主清理链路；确认框和撤销能力应在后续交互轮次再设计。

- 风险：如果删除执行型节点时只杀进程、不先清理监听器和同步定时器，进程关闭事件仍可能晚到并回写到已删除节点。
  当前缓解：宿主删除路径必须先取消定时器、移除监听器、从会话映射里删除，再终止进程。

- 风险：如果键盘删除在输入控件聚焦时仍生效，会误删正在编辑的 `Task` / `Note`。
  当前缓解：画布级删除快捷键只在焦点不属于输入、文本域、下拉框、按钮或其他标记为交互元素的目标上时生效。

- 风险：如果继续沿用 `nodes.length + 1` 生成 ID，删除中间节点后会与仍然存在的节点冲突。
  当前缓解：把“下一序号”改为扫描现存节点的数字后缀并取最大值加一。

## 8. 验证方法

至少需要完成以下验证：

1. 四类节点都能从节点头部删除。
2. 选中节点后，在非输入焦点场景下按删除键可以删除该节点。
3. 删除运行中的 `Agent` / `Terminal` 后，不再继续收到该节点的运行时回流。
4. 删除最后一个节点后，reload 仍保持空画布。
5. 删除中间节点后继续新增同类型节点，不出现 ID 冲突。
6. `npm run build` 和 `npm run typecheck` 通过。
