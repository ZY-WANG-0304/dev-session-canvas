# 执行节点缩放交互面设计

## 1. 背景

当前 `Terminal` 与 `Agent` 节点都已经收口为节点内嵌入式会话窗口，且节点必须跟随 React Flow 一起缩放。这是当前产品定义，不是待讨论项。真实的 `xterm.js` 前端也仍直接挂在 React Flow 节点内部；当画布缩放后，鼠标在终端中拖选时，命中的字符区域与视觉区域会出现偏移。

这个问题不是后端 PTY 路线的问题，而是前端坐标系的问题。当前实现里，执行节点跟随 React Flow viewport 一起做 `translate + scale`；但 `xterm.js` 的鼠标命中、选择和字符网格计算仍按自己的逻辑 cell 尺寸进行。两套坐标系一旦不再等比，就会出现“看到的字符在这里，拖选命中却在别处”的体验错误。

## 2. 问题定义

本轮需要回答的问题是：

1. 如何修复缩放后的终端鼠标命中与选择偏移。
2. 如何在修复这个 bug 的同时，保持“终端仍随 React Flow 缩放、主交互仍属于节点”的产品语义。
3. 如何控制实现复杂度，避免为了修一个拖选 bug，直接把运行时表面重构成另一套新架构。

## 3. 目标

- 修复 `Terminal` 与 `Agent` 两类执行节点在画布缩放下的鼠标拖选、点击和命中偏移。
- 保留“终端继续随画布缩放”“主要输入、输出和聚焦行为仍在节点语义内发生”这两条既有产品结论。
- 让修复范围收口为“坐标补偿”，而不是借题扩张成新的默认交互面架构。

## 4. 非目标

- 不在本轮重新讨论 PTY 后端、会话恢复或 `xterm.js` 是否继续使用。
- 不在本轮引入 viewport 外 overlay、默认 drawer/pane 或其他新的主要交互表面。
- 不把这次 bug 修复扩张成新的共享 `xterm` 适配层重构。

## 5. 当前根因

当前 Webview 实现中，`AgentSessionNode` 与 `TerminalSessionNode` 都在节点内部直接创建 `xterm.js` 实例；对应 DOM 会跟随 React Flow viewport 一起缩放。与此同时，`xterm.js` 的鼠标坐标换算依赖元素屏幕矩形和自己的 CSS cell 尺寸。

这意味着：

- React Flow 管的是“节点整体如何在画布里缩放和平移”。
- `xterm.js` 管的是“鼠标在终端字符网格里落到哪一列哪一行”。
- 当真实交互前端继续放在被缩放的节点树里时，这两层会共享视觉结果，但不会天然共享同一套命中语义。
- `xterm.js` 虽然大多数鼠标命中都经过 `MouseService`，但拖选越界滚动和 textarea 定位仍各有一条旁路逻辑，不能只补一处就认为所有交互都会自动好。

因此，这不是纯样式小 bug，而是“当交互前端留在被缩放容器里时，哪些鼠标逻辑必须按 zoom 做坐标补偿”的边界问题。

## 6. 候选方案

### 6.1 方案一：保持节点内缩放，给 `xterm.js` 做鼠标坐标补偿

特点：

- React Flow 节点内部继续保留真实 `xterm.js`，并继续随画布一起缩放。
- 当前 viewport `zoom` 被显式用于鼠标逻辑坐标补偿，而不是用于改变 PTY `cols/rows`。
- 补偿重点落在 `MouseService` 的统一命中入口，并补齐拖选越界滚动与 textarea 定位这两条旁路。

优点：

- 完全保留当前产品表面：节点还是节点内真实终端，且继续跟随画布缩放。
- 不改变 `fitAddon`、PTY `cols/rows`、节点内主交互和当前运行时表面结构。
- 改动范围可明确收口在前端命中逻辑，而不是扩大成新的表面架构。

风险：

- 需要接触 `xterm.js` 的内部服务与私有方法，后续升级依赖时要回归验证。
- 如果只补统一入口而漏掉旁路逻辑，会留下拖选越界滚动或右键定位的残缺问题。
- 当前不抽共享适配层，`Terminal` 与 `Agent` 的补丁会暂时重复两份。

当前判断：

- 这是当前标准路线。
- 前提是显式接受“终端继续随画布缩放”，并把修复目标收口为命中坐标补偿，而不是运行时表面重构。

### 6.2 方案二：把真实交互终端提升到 viewport 外的屏幕坐标 overlay

特点：

- React Flow 节点内部只保留标题、状态、最近输出和空态提示。
- 真正可交互的 `xterm.js` 不再挂在被缩放的 viewport 内，而是挂到 viewport 之外的 `ExecutionOverlayLayer`。
- 当前活动节点被选中或进入交互态时，overlay 按节点在屏幕上的矩形位置贴回节点区域，看起来仍然像“在节点里操作”。

优点：

- 命中、拖选和点击都回到单一屏幕坐标系，能从根上绕开当前 bug。
- 命中语义最干净，后续不用再维护缩放内 `xterm` 的特殊补丁。

问题：

- 它直接违背当前新增的产品约束：终端必须继续随 React Flow 缩放。
- 它会把本轮 bug 修复扩大成运行时表面重构，与当前范围不匹配。

当前判断：

- 不作为当前默认路线。
- 先前研究曾把它视为更长期的结构性方案，但在“终端必须跟随画布缩放”的约束下，这条路线不再适合作为本轮结论。

### 6.3 方案三：进入聚焦模式后切到独立终端面板或 drawer

特点：

- 节点保留预览和状态。
- 用户点击节点后，在固定浮层、底部 drawer 或侧边 pane 中打开真正可交互的终端。

优点：

- 结构简单，状态模型也更容易收口。
- 单一活动 surface 的心智清晰，短期风险最低。

问题：

- 它把主交互从节点中抽走，和当前“节点内真实终端仍应跟随画布缩放”的产品定义冲突。
- 对当前产品来说，这会把执行节点重新降级成“入口卡片 + 外部操作面”。

当前判断：

- 不作为当前默认路线。
- 只有在节点内真实终端路线被验证为不可行时，才值得重新打开讨论。

## 7. 当前结论

当前收敛结论如下：

- 执行节点的真实交互前端继续直接挂在 React Flow 缩放 viewport 内，并继续跟随画布缩放。
- 正式路线选择方案一：在节点内保留真实 `xterm.js`，通过 viewport `zoom` 对鼠标逻辑坐标做补偿。
- 补偿点优先落在 `MouseService` 的统一入口，并补齐拖选越界滚动与 textarea 定位这两条旁路。
- 本轮不引入 overlay、默认 drawer/pane，也不把修复扩张成新的共享 `xterm` 适配层。
- 这条结论同时适用于 `Terminal` 和 `Agent`，因为两者当前都使用相同的 `xterm.js` 节点内运行时表面。

这份文档取代了 2026-04-11 研究阶段在“产品是否允许终端脱离节点缩放”上的旧假设。旧研究保留为历史输入，但不再代表当前实现结论。

## 8. 风险与取舍

- 取舍：接受继续接触 `xterm.js` 私有服务的实现风险，换取最小范围内修复当前命中语义。
  原因：产品已经明确要求终端继续随画布缩放；在这个前提下，当前最小有效修复就是坐标补偿。

- 风险：`xterm.js` 升级后，`MouseService` 或 `SelectionService` 的内部实现可能变化，导致补丁失效。
  当前缓解：把当前补偿范围、依赖路径和验证口径写进正式文档，并要求后续升级时回归验证缩放交互。

- 风险：如果只补 `MouseService` 而忽略拖选越界滚动和 textarea 定位，用户仍会在局部交互里看到偏差。
  当前缓解：把这两条旁路显式列入当前修复范围，而不是假设统一入口天然覆盖全部路径。

- 风险：当前不抽共享适配层意味着 `Terminal` 与 `Agent` 的补丁逻辑会暂时重复。
  当前缓解：本轮显式接受这笔技术债，后续若再出现更多同类补丁，再单独开设计/实现收口共享层。

## 9. 验证方法

至少需要完成以下验证：

1. 在 `0.4`、`1.0`、`1.8` 三档缩放下，终端拖选与实际字符区域对齐。
2. 在同三档缩放下，点击、双击、链接 hover/click 与应用侧鼠标上报不再偏移。
3. 在缩放后，拖选超出终端边界时，自动滚动方向和速度不出现明显错乱。
4. 右键与 Linux middle-click 的 textarea 定位不会因为缩放落到明显错误位置。
5. `Terminal` 与 `Agent` 两类节点都完成同类修复，而不是只修其中一种。

当前状态：

- `Playwright harness` 已在 `zoom=1.6` 下覆盖 `Terminal` 与 `Agent` 的拖选对齐、拖选越界滚动边界，以及右键 textarea 定位。
- `Extension Development Host` 中的人工验证也已完成，当前文档对应方案可视为已验证。
