# 画布关系连线设计

## 1. 背景

当前 Dev Session Canvas 已经把 `Agent`、`Terminal`、`Note` 放回同一张画布，但节点之间的依赖、引用和协作链路仍停留在隐式状态：

- 用户可以并排看到多个节点，却无法在画布上直接表达这些节点之间的关系。
- 关系如果只存在于用户脑中或聊天上下文里，就无法和画布状态一起持久化。
- 当文件活动会自动派生 `Agent -> 文件` 关系时，如果没有统一连线模型，自动边和手工边就会分裂成两套系统。

## 2. 问题定义

本设计文档聚焦三个问题：

1. 通用连线应如何进入共享状态模型，而不是只留在 Webview 局部状态。
2. 连线创建、编辑、删除和持久化的交互边界应如何收口。
3. 手工连线与文件活动派生的自动连线应如何共存，而不互相污染事实来源。

## 3. 目标

- 给所有画布节点提供可持久化、可编辑、无预设语义的通用连线。
- 让连线成为宿主权威状态的一部分，并在 reload 后恢复。
- 复用同一套 edge 模型承载手工边和自动边，同时保持事实来源分层清晰。

## 4. 非目标

- 不在本轮为连线引入业务语义、关系类型枚举或校验规则。
- 不在本轮引入更复杂的白板自动布局、连线整理或关系图算法。
- 不把文件活动自动生成的 `Agent -> 文件` 关系线开放成用户可任意编辑的业务关系。

## 5. 候选方案

### 5.1 连线只存在于 Webview 局部状态

特点：

- 前端改动快，不需要先动共享协议。
- 可以快速做出可拖拽的视觉效果。

不选原因：

- reload、宿主持久化和多次消息同步后，连线无法成为可信状态。
- 自动边如果只在宿主重建、手工边如果只在 Webview 保存，会形成两套并行真相。
- 与仓库“先定义谁拥有长期状态”的工作流要求冲突。

### 5.2 连线进入共享协议并由宿主持久化

特点：

- 宿主掌握权威状态，Webview 只负责交互与呈现。
- 手工边和自动边可共享同一套 `edge` 结构与渲染路径。

选择原因：

- 能让连线与节点一起进入画布正式状态模型。
- 便于在 `CanvasPanelManager` 中统一处理持久化、恢复和节点删除后的边清理。
- 为文件活动派生的自动边提供同构承载，而不需要在 UI 层发明第二套关系系统。

## 6. 风险与取舍

- 取舍：采用固定四向锚点，而不是自由吸附点。
  原因：第一轮目标是把关系表达和持久化走通；固定锚点更容易保证交互可预测、状态可序列化。

- 风险：文件活动派生边如果继续完全由宿主重建，又要允许用户编辑 / 删除 / 重接，就会出现“改完立刻被重建打回”的伪交互。
  当前缓解：宿主保留自动边事实来源，但引入持久化覆盖与屏蔽状态；一旦用户修改某条自动边，后续投影优先采用覆盖结果，而不是重新暴露只读边。

- 风险：连线删除与节点删除都依赖选中态，若键盘行为不清晰，容易误删节点。
  当前缓解：`Delete` 键只删除当前选中的连线；节点删除仍走现有节点删除语义，不复用 edge 快捷路径。

## 7. 正式方案

### 7.1 共享协议新增正式 edge 模型

`src/common/protocol.ts` 扩展画布权威状态，新增 `CanvasEdgeSummary`，至少包含以下信息：

- 连线 ID
- 源节点 / 目标节点
- 源锚点 / 目标锚点
- 箭头模式
- 标签文本
- 是否为自动维护边

其中的核心规则是：

- 手工边进入持久化状态。
- 自动边复用同一结构，但事实来源不是用户编辑，而是宿主派生逻辑。
- Webview 不能把本地 edge 当作真相，所有变更都必须回传宿主。

### 7.2 `CanvasPanelManager` 是连线权威入口

`src/panel/CanvasPanelManager.ts` 负责：

- 接收 Webview 回传的手工连线创建、更新和删除消息。
- 把手工连线并入 workspace 绑定画布状态并持久化。
- 在节点删除、reset 或恢复时清理失效边。
- 合并手工边与自动边后，再把完整边集合回传 Webview。

这里的关键边界是：

- 宿主保存的是手工边和文件活动权威状态。
- 自动边由宿主根据文件活动引用重建，不直接信任旧自动边快照。

### 7.3 Webview 只负责交互与呈现

`src/webview/main.tsx` 使用 React Flow edge 模型承接交互，但不拥有最终状态。Webview 负责：

- 在所有可见节点边缘渲染四向 handles。
- 处理拖拽建边、边选中、选中态轻量编辑台、双击标签原位编辑和连线两端重接。
- 将连线的创建、箭头模式修改、颜色修改、标签修改、重接和删除动作统一回传宿主。

`src/webview/styles.css` 保持所有连线走同一套视觉语言：

- 默认连线颜色与拖拽中的预览连线共用同一默认 token。
- 选中与 hover 不通过主线换色区分，而是通过额外 outline、轻量编辑台和端点重接 handles 提示当前交互焦点。
- 端点重接 handles 默认保持透明热区，不额外渲染显眼圆形把手。
- 连线标签采用贴线轻底片，而不是长期常驻的重胶囊控件。

连线编辑交互在 Webview 中遵守以下规则：

- 任意连线在被选中时都显示连线上方的轻量编辑台；右键菜单不再承担连线编辑主路径。
- 属性编辑动作统一收敛到这条上浮轻量编辑台；连线本体不再常驻承担属性操作入口。
- 轻量编辑台沿用 VSCode Workbench token 与当前仓库已引入的 `codicon`，提供箭头模式切换、颜色切换、编辑标签和删除连线四个动作；颜色能力参考 Obsidian 的“默认色 + 6 个预设色”。
- “默认颜色”属于显式颜色切换动作，而不是“什么都不做”；用户切回默认色时，宿主需要清空持久化颜色覆盖。
- 标签默认以贴线轻底片显示，强调“这是连线文本”，而不是独立悬浮卡片。
- 双击连线或连线标签时，标签输入框直接出现在连线标签位置；`Enter` 提交，`Escape` 取消，失焦按当前输入提交。
- 未命名连线在非编辑态不显示“未命名关系”之类的占位胶囊，避免制造“可见但不可编辑”的假象。
- 选中连线后，React Flow 的端点 updater handles 会出现在连线两端，允许用户像 Obsidian Canvas 一样重新调整连线关系。
- 连线允许创建和保留自环；当源节点与目标节点相同时，Webview 需要用可见的自环 path，而不是退化成零长度曲线。

### 7.4 自动边与手工边分层共存

文件活动视图仍会生成 `Agent -> 文件` 自动边，但这类边在用户侧不再暴露为“另一种只读边”。系统内部遵守以下规则：

- 自动边的存在与否只由文件活动引用决定。
- 用户一旦修改某条自动边，宿主会把当前结果持久化为覆盖边，并同步记录对原自动边的屏蔽状态。
- 用户删除某条自动边时，宿主会记录屏蔽状态，使同一条自动边不会在下一次投影时立刻重新出现。
- 手工边和自动边都可以连接任意当前可见节点，但它们不反向影响文件活动引用事实。

这让“关系表达”与“文件活动投影”共享一套图模型，同时保持事实来源分离，而用户侧不需要学习两套连线规则。

### 7.5 生命周期规则

- reload 后只恢复宿主持久化的手工边，再根据当前 `fileReferences` 补齐自动边。
- 删除节点时，同时清理与之关联的手工边；自动边则由宿主重建逻辑自然消失。
- 若文件活动展示模式切换导致 `file` / `file-list` 自动节点集合变化，自动边随之重建；手工边只对仍然存在的节点保留。

## 8. 验证方法

- Playwright Webview 回归：
  - 创建、编辑、删除手工连线。
  - 连线标签、箭头模式、颜色、自环和重接结果在 UI 中正确显示。
  - 选中连线后出现轻量编辑台，双击后进入原位标签编辑。
  - 选中连线后按 `Delete` 键只删除连线，不误删节点。
- VSCode smoke：
  - 手工连线在 reload 后保持。
  - 删除节点后，相关失效边被正确清理。
  - 文件活动派生边在被用户编辑或删除后，reload 与后续文件视图重建不会立刻丢失覆盖结果。
