# 核心信念

这里记录的是协作方式上的通用信念，而不是具体方案结论。

## 1. 先写清楚问题，再开始实现

如果问题定义、目标、边界和验证方式还没有写清楚，就不应让实现细节代替设计文档。

## 2. 未确认的内容宁可留空

空白比伪确定性更安全。对未确认内容，应明确标记为“待定”“待验证”或“候选方案”。

## 3. 设计应服务于产品主张

任何技术方案都应回到核心问题：它是否真的帮助用户在多 Agent 协作中看清全局。

## 4. 计划必须晚于设计

执行计划应来自已经被写明的问题、范围和候选方案，而不是倒逼出方案本身。

## 5. 文档是事实来源

如果某个结论只存在于实现细节、口头讨论或参考产品印象里，就等于它还没有被正式确认。

## 6. 平台支持必须和验证证据一起记录

代码路径已经接通，不等于平台支持已经完成。只要缺少对应平台的人工验证或可靠证据，就必须明确写成“验证中”或“待验证”。

## 7. 不属于空间关系的信息应优先离开画布

当某个按钮、说明或状态并不需要依赖节点位置一起理解时，应优先放到 VSCode 原生侧栏、命令入口或工具栏，而不是继续占用画布固定角落。画布应优先留给节点本体、关系和导航。

## 8. 主工作面应保留稳定回路

如果一个全局主工作面必须长期和用户的高频编辑路径争抢同一宿主区域，就不应把“切回去的成本”默认转嫁给用户。只要 VSCode 提供多个合适的宿主承载面，设计就应优先考虑让主工作面保留稳定、可预测的返回路径。

## 9. 先定义谁拥有长期运行时

如果产品要承诺“关闭画布甚至关闭 VSCode 后，会话仍继续存在”，就必须先写清楚谁拥有进程、谁负责日志、谁负责清理。不要把“快照恢复”误写成“真实进程延续”。

## 10. 第一版默认追求尽量完整实现

第一版不应默认被设计成“只证明可行”的半链路。只有在 blocker、外部依赖边界或暂不支持的平台已经被明确记录时，才允许把能力后移；否则应尽量完整实现产品主张，而不是主动把关键链路拆到后续版本。
