# 核心信念

这里记录的是协作方式上的通用信念，而不是具体方案结论。

## 1. 先写清楚问题，再开始实现

如果问题定义、目标、边界和验证方式还没有写清楚，就不应让实现细节代替设计文档。

## 2. 未确认的内容宁可留空

空白比伪确定性更安全。对未确认内容，应明确标记为“待定”“待验证”或“候选方案”。

## 3. 设计应服务于产品主张

任何技术方案都应回到核心问题：它是否真的帮助用户在多 Agent 协作中看清全局。

## 4. 计划必须晚于设计

执行计划应来自已经被写明的问题、范围和候选方案，而不是倒逼出方案本身。

## 5. 文档是事实来源

如果某个结论只存在于实现细节、口头讨论或参考产品印象里，就等于它还没有被正式确认。

## 6. 平台支持必须和验证证据一起记录

代码路径已经接通，不等于平台支持已经完成。只要缺少对应平台的人工验证或可靠证据，就必须明确写成“验证中”或“待验证”。
