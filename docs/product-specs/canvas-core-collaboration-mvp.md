# 画布核心协作 MVP 规格

## 1. 用户问题

在 VSCode 中并行使用多个 Agent、终端、任务和笔记时，开发者的上下文容易分散在多个 panel、editor、标签页和临时记录里，导致以下问题：

- 很难快速看清当前有哪些执行单元正在活动
- 很难理解终端、任务、笔记和 Agent 之间的关系
- 切到 VSCode 原生终端或编辑器后，容易丢失回到全局视角的路径
- 在 reload、重开窗口或远程工作区场景下，关键上下文容易断裂

## 2. 目标用户

当前规格优先服务已经在 VSCode 中进行 AI 协作或 Harness Engineering 的开发者。用户通常已经习惯 VSCode 原生终端、编辑器和 workspace 语义，但缺少一个能把多执行单元放回同一空间中理解的主视图。

## 3. 核心用户流程

1. 用户在一个 VSCode workspace 中打开画布。
2. 用户看到一张可平移、缩放的无限画布；初始状态允许为空，而不是强制显示示例节点。
3. 用户可以在画布上创建和删除四类对象：Agent、终端、任务、笔记。
4. 用户可以直接在执行型对象节点内进行最小交互，而不必始终依赖外部 inspector。
5. 用户可以查看这些对象的标题、位置、状态与基本关系。
6. 用户可以删除不再需要的对象；如果删除的是正在运行的执行型对象，会话也会同步结束并退出对象图。
7. 用户可以直接在画布中的 `Terminal` 节点里使用终端；当需要进入其他 VSCode 原生能力时，再从画布对象跳转过去。
8. 当用户完成局部操作后，可以明确地回到画布，并继续从全局视角理解当前进度。
9. 当 Webview 隐藏、reload 或窗口重开后，系统至少能恢复关键上下文，而不是回到完全空白状态。

## 4. 在范围内

- 一个运行在 VSCode Editor Group 中的主画布入口
- 画布的基础导航能力：平移、缩放、查看对象
- 四类核心对象的最小创建、删除与展示能力：
  - Agent
  - 终端
  - 任务
  - 笔记
- 对象的基础状态展示，例如标题、类型、位置、最近状态摘要
- Agent 对象的最小真实 backend 主路径：
  - 用户可选择 `Codex` 或 `Claude Code`
  - 用户新建 Agent 节点后可先切换 provider，再显式启动真实 CLI 会话；系统按当前 provider 启动对应 CLI
  - 用户直接在 Agent 节点内和 CLI 会话交互，而不是通过独立 prompt 调用
  - 节点至少保留连续输出、活动态、关闭态、失败态和最近输出摘要
- 终端对象的最小真实主路径：
  - 用户新建 `Terminal` 节点后，系统默认自动启动一次终端会话
  - 用户可直接在节点内启动和使用嵌入式终端
  - 节点至少保留连续输出、活动态、关闭态和最近输出摘要
  - Webview 隐藏或重新显示后，仍可重新附着到当前活跃终端
- Task 对象的最小真实编辑主路径：
  - 用户可在节点内编辑标题、状态、任务描述和负责人
- Note 对象的最小真实编辑主路径：
  - 用户可在节点内编辑标题和正文
- 画布关键状态恢复：
  - 至少包含对象图恢复
  - 至少包含最后一次打开画布的基本上下文恢复
- 空画布默认态与重置语义：
  - 宿主默认状态允许没有任何节点
  - 用户执行重置后，应回到空画布，而不是自动生成示例 `Task` / `Note`
- 在未信任 workspace 或能力不可用时，给出明确退化反馈

## 5. 不在范围内

- 独立 app 的 workspace 管理、启动器和多 workspace 组织
- 像素级复刻参考产品全部终端细节
- 跨平台完备且长期稳定的终端后端矩阵
- provider 原生 TUI 的像素级复刻
- 多人实时协作
- 复杂分组、嵌套、自动布局和高级白板能力
- 完整 Agent backend 生态或多 provider 深度接入
- 导入导出格式、分享机制和跨项目同步
- 像素级复刻参考产品视觉与交互

## 6. 关键对象与状态

### 画布

- 打开状态
- 当前对象集合
- 相机位置与缩放
- 当前选中对象

### Agent 对象

- 标题
- 状态
- 当前 provider
- 节点内的嵌入式 CLI 会话窗口
- 最近一步摘要
- 最近输出摘要或退出信息
- 与其他对象的引用关系
- 启动、停止与重启控制

### 终端对象

- 标题
- 关联的宿主终端会话引用
- 节点内终端前端与连续输出
- 活跃状态摘要
- 最近输出摘要
- shell 路径、cwd 与最近退出信息

### 任务对象

- 标题
- 状态
- 负责人或关联 Agent

### 笔记对象

- 标题
- 内容摘要
- 与其他对象的引用关系

### 恢复相关状态

- 与 workspace 绑定的对象图和关键上下文
- Webview 局部 UI 状态与宿主权威状态的边界
- 在 reload 或重开窗口后，哪些状态应恢复、哪些允许作为临时态丢失

## 7. 验收标准

- 用户可以从 VSCode 中打开主画布，并在同一工作面中看到无限画布。
- 用户可以创建 Agent、终端、任务、笔记四类对象，并看到它们出现在画布上。
- 用户在重置宿主状态后会回到空画布，而不是看到自动生成的示例节点。
- 用户可以删除 Agent、终端、任务、笔记四类对象；删除运行中的执行型对象时，会话会同步结束，且对象在 reload 后不会无声恢复。
- 用户可以在多对象同时存在时识别每个对象的基本状态，而不必逐个打开独立 panel 才理解当前情况。
- 左下角缩放控件与基础导航区域不应被固定说明浮层遮挡。
- 右侧固定区域不再保留选中节点的辅助概况区；当前主路径需要暴露的信息应直接留在节点本体中，原先右侧承载的 richer metadata 当前不要求继续展示。
- 连续新增不同类型节点时，新节点默认不与已有节点重叠，并应优先落在用户当前视口附近。
- 用户可以直接在 Agent 节点内使用真实 CLI 会话窗口，而不必依赖右侧详情区或独立 prompt 组件才能与 Agent 交互。
- 用户可以直接在终端对象节点内输入命令并看到实时输出，而不必再跳到画布外的终端面板。
- 当 Webview 被隐藏后重新显示时，当前活跃终端仍可重新附着，而不是无声丢失。
- 当 Webview 被隐藏、重新显示或 reload 后，画布不会无声丢失所有对象状态。
- 当窗口重开或扩展重启后，系统至少能恢复关键上下文，而不是回到全新空白页。
- 在 Restricted Mode 或相关能力不可用时，执行型操作会被禁用或退化，并给出明确反馈。

## 8. 开放问题

- 终端对象的“最近输出摘要”在第一版应做到什么粒度，才能既有价值又不引入过高实现复杂度。
- 当前 `node-pty` 后端在 macOS、Windows、Remote SSH / Codespaces 下的验证矩阵应如何补齐，才能把“代码已接通”升级成“平台已支持”。
- 第一版是否需要显式连接线，还是先依靠对象内引用信息表达关系。
- 关键上下文恢复的最小集合应以什么口径界定，才能既满足可靠性要求，又不把所有临时 UI 状态都强行持久化。
