# SECURITY

本文件用于说明本仓库的安全漏洞如何被私下报告、当前支持范围、维护者的响应预期，以及漏洞披露原则。

它不负责描述系统架构中的安全设计边界。系统安全与隐私约束应记录在 `ARCHITECTURE.md` 与 `docs/design-docs/` 的具体设计文档中。

## 当前状态

当前仓库仍处于前期定义阶段，尚未发布任何稳定版本，也尚未建立对外公开的专用安全报告渠道。

这意味着两件事：

- 当前不能把普通公开 issue、MR 或讨论区当作安全漏洞入口。
- 在仓库对外发布、开始接收外部部署或外部贡献前，必须补齐可用的私密报告渠道，并同步更新本文件。

## 当前支持范围

当前尚无已发布的版本线。以 2026-03-28 的仓库状态，支持范围按下表理解：

| 版本 / 分支 | 是否接收安全修复 | 说明 |
| --- | --- | --- |
| `main` 最新开发线 | 有限支持 | 当前仓库只有未发布开发线；如出现确认的安全问题，应在最新开发线上处理 |
| 历史提交、临时分支、个人派生修改 | 不承诺 | 不视为受支持发布版本 |
| 已发布稳定版本 | 不适用 | 当前尚未发布稳定版本 |

如果后续出现正式版本号、长期维护分支或停止支持的旧版本，必须先更新本节，再对外发布。

## 如何报告漏洞

如果你认为发现了安全漏洞，请优先通过私密渠道报告，不要直接公开提交 issue、MR、讨论帖或在其他公开场合披露细节。

当前仓库尚未建立对外公开的专用安全邮箱或漏洞表单，因此本节目前只有以下约束成立：

- 不接受公开 issue 作为安全问题入口。
- 如果你已经拥有仓库维护者的私密联系方式，应优先通过该私密渠道报告。
- 如果你没有任何已知的私密联系渠道，请先不要公开披露；仓库维护者需要先补齐正式渠道后，才能把本文件视为完整可执行的外部安全政策。

这不是理想状态。仓库在进入对外使用前，至少需要补齐以下其一：

- 专用安全邮箱
- 私密漏洞报告表单
- 受控的安全反馈平台
- 明确的内部值班 / 维护者私密入口

## 报告中应包含的信息

无论后续使用哪种私密渠道，安全报告都建议尽量包含以下信息：

- 受影响的分支、提交或版本
- 运行环境，例如本地 VSCode、Remote、Codespaces 或其他宿主形态
- 复现步骤
- 影响范围与预期风险
- PoC、截图、日志或最小复现样例
- 是否涉及凭证、文件访问、终端执行、远程代码执行、权限绕过或敏感数据泄露
- 你认为是否已经存在公开暴露风险

## 响应预期

当前仓库尚未建立正式对外响应承诺，因此这里不写虚假的 SLA。

在正式安全渠道建立前，可以先把响应原则固定为：

- 安全问题应优先于普通功能 bug 处理
- 维护者在确认收到报告后，应先判断是否需要限制传播范围
- 高危问题应优先评估缓解措施、修复路径和是否需要临时禁用相关能力
- 对外承诺的具体响应时间，必须在专用渠道建立时一并补齐，并保持现实可执行

## 披露原则

在维护者完成调查、风险评估与修复协调前，不应公开披露漏洞细节。

当前披露原则如下：

- 默认采用私下沟通优先
- 修复可用前，不应公开发布可复现细节、PoC 或利用路径
- 如果后续需要公开披露，应由维护者与报告者协调时机
- 是否发布 security advisory、release note 或致谢条目，当前待正式发布流程建立后再补齐

## 受理范围与非受理范围

### 范围内

- 本仓库中的 VSCode 扩展代码、配置与发布产物
- 本仓库维护的宿主集成、Webview、终端桥接、Agent 适配与状态持久化实现
- 由于本仓库代码或默认配置直接导致的权限绕过、远程代码执行、敏感数据泄露、越权访问或信任边界失效

### 默认不在范围内

- 普通功能缺陷、体验问题或性能问题
- 尚未实现能力对应的“理论风险”
- 仅存在于上游第三方依赖、但尚未证明会在本仓库默认使用路径中造成可利用风险的问题
- 社会工程学测试、钓鱼、物理接触攻击
- 未经明确授权的破坏性压力测试、资源滥用或 DoS 测试
- 仅依赖本地开发者自行关闭安全设置、绕过信任边界或修改源码后才成立的问题

## 普通问题与安全问题的分流

- 安全漏洞：走本文件定义的私密报告路径
- 普通 bug、需求或使用问题：走常规 issue / MR / 团队协作流程

如果一个问题同时涉及功能错误和安全影响，应先按安全问题处理，再决定哪些细节可以公开同步。

## 维护要求

以下任一情况发生时，应更新本文件：

- 建立或变更专用安全联系方式
- 发布首个正式版本
- 停止维护旧版本或新增长期维护分支
- 确认新的披露策略、响应时间或致谢规则

本文件的及格标准不是“证明项目已经足够安全”，而是让发现漏洞的人知道应如何私下报告、维护者如何处理，以及什么内容当前仍未建立。
