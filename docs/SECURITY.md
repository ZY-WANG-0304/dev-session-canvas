# SECURITY

本文件说明安全漏洞的私下报告方式、支持范围、维护者响应预期与披露原则。

系统架构中的安全设计边界不在本文件范围内，应记录在 `ARCHITECTURE.md` 与 `docs/design-docs/` 中。

## 状态

产品已进入公开 `Preview` 阶段，尚未发布稳定版本。当前仓库按公开 `Preview` 口径持续迭代，并已建立专用安全报告邮箱。

这意味着：

- 不能将公开 issue 或讨论区作为安全漏洞入口。
- 只承诺处理最新主线 / 预览版中的安全问题，不覆盖历史预览包、临时分支或个人派生修改。

## 支持范围

尚无稳定版本线。支持范围如下：

| 版本 / 分支 | 是否接收安全修复 | 说明 |
| --- | --- | --- |
| `main` 最新开发线 / 最新公开预览版 | 有限支持 | 仅支持最新主线与最新预览版；确认的安全问题优先在最新代码线上修复 |
| 历史预览包、历史提交、临时分支、个人派生修改 | 不承诺 | 不视为受支持发布版本 |
| 已发布稳定版本 | 不适用 | 尚未发布稳定版本 |

如后续出现正式版本号、长期维护分支或停止支持的旧版本，须先更新本节再对外发布。

## 如何报告漏洞

发现安全漏洞后，请通过私密渠道报告，不要公开提交 issue、MR、讨论帖或在其他公开场合披露细节。

专用安全邮箱：

- `wzy0304@outlook.com`

报告要求：

- 不接受公开 issue 作为安全问题入口。
- 请将安全问题发送至上述邮箱，并在邮件标题中标注 `Security`、受影响版本或问题摘要。
- 若问题已存在公开暴露风险，请在邮件中显式说明，以便维护者优先分级。

## 报告中应包含的信息

安全报告建议尽量包含以下信息：

- 受影响的分支、提交或版本
- 运行环境（本地 VS Code、Remote、Codespaces 或其他宿主形态）
- 复现步骤
- 影响范围与预期风险
- PoC、截图、日志或最小复现样例
- 是否涉及凭证、文件访问、终端执行、远程代码执行、权限绕过或敏感数据泄露
- 是否已存在公开暴露风险

## 响应预期

- 安全问题优先于普通功能 bug 处理
- 维护者在 `3 个工作日内` 确认收到报告
- 维护者在 `5 个工作日内` 完成初步分级，判断是否属于安全问题及是否需要限制传播范围
- 高危或疑似已被利用的问题会加急处理；修复完成前，维护者可能先同步缓解措施、临时禁用建议或风险告警
- 不承诺统一的最终修复 SLA；修复窗口取决于严重度、影响范围、是否已有利用迹象及发布节奏

## 披露原则

在维护者完成调查、风险评估与修复协调前，不应公开披露漏洞细节。

- 默认私下沟通优先
- 修复可用前，不应公开可复现细节、PoC 或利用路径
- 如需公开披露，应由维护者与报告者协调时机
- 是否发布 security advisory、release note 或致谢条目，待正式发布流程建立后补齐

## 受理范围与非受理范围

### 范围内

- 本仓库中的 VS Code 扩展代码、配置与发布产物
- 本仓库维护的宿主集成、Webview、终端桥接、Agent 适配与状态持久化实现
- 因本仓库代码或默认配置直接导致的权限绕过、远程代码执行、敏感数据泄露、越权访问或信任边界失效

### 不在范围内

- 普通功能缺陷、体验问题或性能问题
- 尚未实现能力对应的"理论风险"
- 仅存在于上游第三方依赖、且未证明会在本仓库默认路径中造成可利用风险的问题
- 社会工程学测试、钓鱼、物理接触攻击
- 未经明确授权的破坏性压力测试、资源滥用或 DoS 测试
- 仅在关闭安全设置、绕过信任边界或修改源码后才成立的问题

## 问题分流

- 安全漏洞：走本文件定义的私密报告路径
- 普通 bug、需求或使用问题：提交到 GitHub Issues <https://github.com/ZY-WANG-0304/dev-session-canvas/issues>

若问题同时涉及功能错误和安全影响，应先按安全问题处理，再决定哪些细节可以公开。

## 维护要求

以下任一情况发生时，应更新本文件：

- 建立或变更专用安全联系方式
- 发布首个正式版本
- 调整"只支持最新主线 / 预览版"的支持口径
- 停止维护旧版本或新增长期维护分支
- 确认新的披露策略、响应时间或致谢规则

本文件的目标不是证明项目足够安全，而是让发现漏洞的人知道如何私下报告、维护者如何处理，以及哪些机制尚未建立。
