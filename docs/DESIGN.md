# DESIGN

本文件定义 `docs/design-docs/` 的使用方式，以及设计文档的状态与元数据约定。

## 职责边界

- `ARCHITECTURE.md`：提供顶层架构地图、领域边界和稳定接口。
- `docs/DESIGN.md`：定义设计文档工作流、状态和 frontmatter 约定。
- `docs/design-docs/index.md`：作为具体设计文档的注册表。
- `docs/design-docs/*.md`：记录问题定义、候选方案、取舍、正式方案或当前结论，以及验证证据。

## 什么时候写设计文档

当以下任一内容需要被明确时，应新增或更新设计文档：

- 问题定义
- 目标与非目标
- 候选方案比较
- 关键边界划分
- 风险与取舍
- 待验证假设

## 当前约定

- 如果方案尚未确认，文档可以只写问题、约束和候选项。
- 如果结论尚未形成，明确写“待定”即可。
- 不要把倾向、直觉或草案写成已接受结论。
- 每份具体设计文档必须使用 YAML frontmatter。
- `decision_status: 已选定` 的文档，应使用 `正式方案` 章节收口当前正式口径，而不是继续停留在泛化的“当前结论”。
- `正式方案` 应直接写出明确方案，而不是只给一句摘要式口号。
- 涉及模块与代码锚点不应孤立悬挂在方案之外；应直接写进 `正式方案` 的说明中，让读者能从方案描述直接定位主要实现落点。
- 如果仅写路径、模块、类型或函数名仍不足以支撑定位，`正式方案` 可按需补充对应代码位置与行号。
- `正式方案` 不能只停留在“背景、问题、候选方案、决策过程”的回顾；应根据方案实际情况承载必要的方案细节。
- 当前仓库仍存在尚未按本规范迁移的历史 `已选定` 文档。过渡期内，默认要求“新写或本次改动到的 `已选定` 文档”使用 `正式方案` 章节；未改动的历史文档可暂时保留 `当前结论` 写法，但应视为待迁移技术债，而不是已完成规范收口。
- `decision_status: 待探索` 或 `比较中` 的文档，可以继续使用 `当前结论` 或 `当前判断`，但不得伪装成最终方案。

## Frontmatter 约定

每份具体设计文档都必须包含以下字段：

```yaml
---
title: 文档标题
decision_status: 待探索
validation_status: 未验证
domains: []
architecture_layers: []
related_specs: []
related_plans: []
updated_at: 2026-03-27
---
```

- `title`：文档标题，应与正文标题一致。
- `decision_status`：决策状态，必须使用固定枚举。
- `validation_status`：验证状态，必须使用固定枚举。
- `domains`：关联问题域；应与 `ARCHITECTURE.md` 中的领域划分保持一致。
- `architecture_layers`：关联层或运行时边界；应与 `ARCHITECTURE.md` 保持一致。
- `related_specs`：关联产品规格路径。
- `related_plans`：关联执行计划路径。
- `updated_at`：最后更新时间，使用 `YYYY-MM-DD`。

## 状态枚举

### 决策状态

- `待探索`
- `比较中`
- `已选定`
- `已废弃`

### 验证状态

- `未验证`
- `验证中`
- `已验证`
- `验证失败`

## 索引同步约定

- `docs/design-docs/index.md` 必须登记每份具体设计文档的状态。
- 索引状态应与具体设计文档 frontmatter 保持一致。
- 如果两处不一致，以具体设计文档 frontmatter 为准，并尽快同步索引。

## 推荐结构

1. 背景
2. 问题定义
3. 目标
4. 非目标
5. 候选方案
6. 风险与取舍
7. 正式方案（若已选定）或当前结论（若仍在比较/探索）
8. 验证方法

## 正式方案章节约定

当文档 `decision_status` 为 `已选定`，且该文档是新写或本次正在改动的正式设计文档时，`正式方案` 章节至少应包含以下信息：

1. 明确的方案说明
2. 适用范围与边界
3. 核心规则与不变量

方案说明本身应优先写入仓库内的真实路径、模块、类型或函数名，帮助协作者从设计结论直接定位到主要落点。例如：

- `src/panel/CanvasPanelManager.ts`：宿主权威状态与消息编排入口
- `src/common/protocol.ts`：共享消息与节点模型
- `src/webview/main.tsx`：Webview 侧交互与呈现

当仅靠上述命名仍不足以避免歧义时，可以继续补充更细的代码锚点，例如关键函数、代码片段所在段落，或必要的行号引用，前提是这些引用直接服务于解释正式方案，而不是脱离方案正文孤立堆砌定位信息。

除上述基础内容外，`正式方案` 还应根据方案实际情况，按需体现必要的结构和内容。常见但非强制的内容包括：

- 数据结构及其关系，例如表结构、实体关系、状态对象关系
- 业务流程图或业务时序图，按用户操作或角色协作维度解释主路径
- 程序流程图或程序时序图，按代码执行、消息流转、状态推进维度解释运行路径
- 接口约定，例如对外公开的方法、消息协议、API、输入输出和错误语义
- 其他有助于把方案说清的内容，例如伪代码、类图、思维导图、泳道图，以及对安全、性能、边界情况、性价比的分析
- 附注，例如补充说明、引用资料、外部约束来源

这些内容不是每份文档都必须全部具备；应由文档作者根据方案复杂度、受众和实现风险，选择真正有助于解释正式方案的部分。原则是：不缺关键细节，也不为了形式堆砌无关图表。
