# 建立终端代理节点

本 ExecPlan 是活文档。推进过程中必须持续更新 `进度`、`意外与发现`、`决策记录` 和 `结果与复盘`。

## 目标与全局图景

把当前 React Flow 画布中的 `Terminal` 节点从展示型占位节点升级为真正的终端代理节点，验证以下最小主路径：

- 画布中可以创建终端节点
- 终端节点可以创建或绑定 VSCode 原生终端会话
- 用户可以从终端节点跳转到真实终端
- 终端会话关闭后，节点状态能回流到画布
- 终端相关状态不会因为扩展 reload 而直接失真

本阶段仍不实现嵌入式终端，只验证“原生终端代理节点”路线是否成立。

## 意外与发现

- 观察：当前画布主线已经具备宿主权威状态、消息桥、serializer 和空间化节点呈现，因此终端代理节点可以直接落在现有主线上。
  证据：前序 `WebviewPanel` 原型与 React Flow 原型都已通过自动化和手动验证。

- 观察：VSCode 本地类型定义已经足够支撑第一版终端代理节点实现，不需要额外引入复杂终端桥接层。
  证据：本地 `@types/vscode` 已提供 `TerminalLocation`、`TerminalEditorLocationOptions`、`onDidCloseTerminal` 等当前所需 API。

## 决策记录

- 决策：第一版终端代理节点使用 VSCode 原生终端 API，而不是嵌入式终端。
  理由：这与设计文档中“原生终端代理节点优先”的收敛一致，且当前更适合验证状态映射、跳转和恢复链路，而不是额外引入 `xterm.js + pty` 风险。
  日期/作者：2026-03-28 / Codex

- 决策：终端默认在编辑器区域 `Beside` 打开，而不是直接固定在终端面板。
  理由：这样更容易保留“画布仍是主工作面”的空间关系，也更便于用户在编辑器组内往返终端与画布。
  日期/作者：2026-03-28 / Codex

## 工作计划

1. 扩展节点模型，给终端节点增加最小元数据。
2. 实现宿主侧终端服务，管理创建、显示、关闭回流与 reload 后的重新匹配。
3. 在 Webview 中增加终端节点详情与动作按钮。
4. 运行构建、类型检查和手动验证。

## 具体步骤

1. 定义终端节点元数据与终端动作消息协议。
2. 在宿主侧维护终端名称与终端节点的映射。
3. 为终端节点增加“创建并显示终端”“显示终端”等动作。
4. 监听终端关闭事件并更新节点状态。
5. 重新打开画布或 reload 后，恢复终端节点与现存宿主终端的匹配状态。

## 验证与验收

本轮至少满足以下条件才算完成：

- `Terminal` 节点不再只是占位卡片，而具备真实宿主动作。
- 用户可以从终端节点创建或显示一个 VSCode 原生终端。
- 终端关闭后，节点状态会变化，而不是停留在假状态。
- `npm run build` 与 `npm run typecheck` 通过。
- 手动验证中至少覆盖：创建终端、显示终端、关闭终端、reload 后恢复状态。

## 幂等性与恢复

- 如果终端对象元数据结构需要调整，应保证旧状态能被归一化处理。
- 如果扩展 reload 后无法保留 `Terminal` 对象引用，应优先通过稳定名称重新匹配，而不是把整条终端代理路线推翻。
- 所有构建和验证命令都应可重复执行。
