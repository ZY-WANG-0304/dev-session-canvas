# OpenCove Extension 架构说明

本文件提供当前项目的顶层架构地图，帮助协作者理解系统边界、核心领域和稳定接口。

本文件刻意保持轻量。它只记录当前已经确认的目标、边界和约束，不提前写死尚未做完设计的实现方案。

## 1. 项目背景

当前项目的目标是在 VSCode 内复刻 OpenCove 的核心产品体验：把 AI Agents、终端、任务和笔记放到同一张无限 2D 画布上，让用户在多 Agent 协作时仍能保持全局视角。

与独立 app 不同，本项目运行在 VSCode 内，天然工作在用户当前打开的 workspace 中。因此，独立 app 里的 workspace 管理能力不属于当前范围。

## 2. 目标

当前阶段的顶层目标如下：

- 在 VSCode 内提供一张可平移、缩放和组织对象的无限 2D 画布。
- 在同一画布上承载至少四类核心对象：Agent、终端、任务、笔记。
- 让用户能在多对象并行活动时持续看清上下文、关系和整体进度。
- 让 VSCode 的 workspace 上下文自然进入该体验，而不是再造一套独立 workspace 管理系统。

## 3. 非目标

当前顶层架构不把以下内容视为必须成立的能力：

- 独立于 VSCode 的桌面应用能力。
- 独立 app 形态下的 workspace 创建、切换和管理。
- 在当前阶段追求对 OpenCove 的像素级复刻。
- 在正式设计完成前提前锁定具体 AI provider、存储引擎或画布渲染细节。

## 4. 约束

当前明确的约束如下：

- 宿主约束：系统必须运行在 VSCode 扩展机制内。
- 产品约束：当前优先复刻核心协作体验，而不是补齐外围管理能力。
- 上下文约束：workspace 语义直接复用 VSCode 提供的上下文。
- 复杂度约束：在灵活性与简单性冲突时，优先简单。
- 文档约束：关键设计决策必须进入正式文档，复杂任务优先通过 `ExecPlan` 推进。

## 5. 领域划分

当前项目可先按以下领域理解：

`VSCode 集成域`：负责扩展激活、命令注册、Webview 生命周期、终端 API、workspace 访问和宿主事件接入。

`画布交互域`：负责无限画布本身，包括平移、缩放、布局、选择、连接关系和对象可视化。

`协作对象域`：负责定义 Agent、终端、任务、笔记这些对象的共享模型、状态、元数据和彼此关系。

`执行编排域`：负责把协作对象和实际执行能力连接起来，例如 Agent 生命周期、终端会话、任务状态推进等。

`项目状态域`：负责与当前 workspace 相关的布局、会话状态和最小必要的持久化。

## 6. 运行时边界与分层

当前更适合把系统理解为四个核心边界，而不是一条严格的线性分层：

`宿主集成层`：运行在 VSCode Extension Host 中，处理 VSCode API、命令、终端、文件和 Webview 容器。

`画布呈现层`：运行在 Webview 中，负责用户可见的画布、节点和交互反馈。

`共享模型与编排层`：负责定义跨宿主与 Webview 共享的对象模型、动作、消息载荷和状态流转规则。

`适配与基础设施层`：负责日志、持久化、配置，以及把共享模型连接到具体 Agent/终端/任务实现的适配器。

当前允许的依赖方向是：

- 宿主集成层和画布呈现层都可以依赖共享模型与编排层。
- 适配与基础设施层可以为共享模型与宿主层提供能力。
- 共享模型与编排层不应直接依赖 VSCode API、DOM 或具体 AI provider。
- Webview 不应直接越过消息边界调用宿主能力。

## 7. 关键稳定接口

当前应优先保持稳定的接口如下：

`Webview <-> Extension Host 消息边界`：传递用户动作、对象更新、同步状态和宿主能力请求；消息协议应避免泄漏过多 VSCode 或 UI 实现细节。

`共享对象模型 <-> 执行适配器边界`：画布对象应依赖抽象能力，而不是直接依赖具体 Agent runtime、终端实现或第三方服务。

`共享对象模型 <-> 持久化边界`：与 workspace 绑定的状态应能序列化、恢复和演进，但当前不预设最终存储格式。

## 8. 横切关注点

质量：核心判断标准不是对象种类有多少，而是用户是否还能看清全局、定位当前工作、恢复上下文。

可靠性：任何导致对象状态、终端上下文、任务进度或笔记内容无声丢失的设计都不可接受。

安全与隐私：扩展可能触达代码、终端、任务和 AI 服务，相关权限、日志和凭证边界必须显式记录。

性能：画布需要支撑多对象同时存在，不能因为对象增多就迅速失去可读性或交互流畅度。

## 9. 风险与未知项

当前仍需后续设计文档澄清的关键未知项包括：

- Webview 中如何表达终端体验，以及与 VSCode 原生终端的关系边界。
- Agent 运行时是完全复用外部能力，还是在扩展内建立新的编排层。
- 画布对象之间的连接、分组和导航模型如何定义，才能既保留全局感，又不造成噪音。
- 哪些状态需要持久化到 workspace，哪些只应作为临时会话态存在。

## 11. 验证方式

后续顶层架构是否成立，应至少通过以下方式验证：

- 在 VSCode 内完成一条最小主路径：打开画布、创建核心对象、查看状态变化。
- 在包含多个 Agent、终端、任务和笔记的场景中，验证用户仍能保持全局可读性。
- 检查具体设计文档、产品规格和实现对本文件中的边界描述是否持续一致。
